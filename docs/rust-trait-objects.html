Trait Objects & Dynamic Dispatch

Rust Trait Objects Dynamic Dispatch

Rust gives you two ways to work with trait-bounded types: static dispatch via monomorphization (the compiler generates a concrete copy per type) and dynamic dispatch via trait objects (the concrete type is resolved at runtime through a vtable pointer). Static dispatch is faster and is the default; dynamic dispatch sacrifices a tiny amount of performance for genuine runtime polymorphism. Knowing when to use each is essential for designing flexible, ergonomic APIs.

Static vs Dynamic Dispatch

When you write fn foo(x: &impl Trait) or fn foo<T: Trait>(x: &T), the compiler sees every concrete type foo is called with and generates a separate, fully optimized version of the function for each one. This is monomorphization and results in machine code with no indirection — equivalent to hand-writing the function for each type. The cost is longer compile times and larger binaries.

When you write fn foo(x: &dyn Trait), the compiler does not know the concrete type at compile time. At runtime, the value is represented as a fat pointer: a pair of pointers, one to the data and one to a vtable (virtual dispatch table). The vtable is a struct of function pointers generated by the compiler for each concrete type that implements the trait. Calling a method on a dyn Trait dereferences the vtable pointer and performs an indirect function call — roughly the cost of one cache miss per call.

Side-by-Side Comparison

The same draw operation implemented with static dispatch (impl Trait) and dynamic dispatch (dyn Trait), showing when each is required.

use std::fmt;

trait Shape: fmt::Display {
    fn area(&self) -> f64;
    fn perimeter(&self) -> f64;
}

#[derive(Debug)] struct Circle { radius: f64 }
#[derive(Debug)] struct Rectangle { w: f64, h: f64 }
#[derive(Debug)] struct Triangle { a: f64, b: f64, c: f64 }

impl fmt::Display for Circle {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result { write!(f, "Circle(r={})", self.radius) }
}
impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius }
    fn perimeter(&self) -> f64 { 2.0 * std::f64::consts::PI * self.radius }
}

impl fmt::Display for Rectangle {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result { write!(f, "Rect({}x{})", self.w, self.h) }
}
impl Shape for Rectangle {
    fn area(&self) -> f64 { self.w * self.h }
    fn perimeter(&self) -> f64 { 2.0 * (self.w + self.h) }
}

// --- Static dispatch: monomorphized, inlined, zero virtual call overhead ---
fn print_shape_static(s: &impl Shape) {
    println!("{s}: area={:.2}, perimeter={:.2}", s.area(), s.perimeter());
}

// --- Dynamic dispatch: single function, works with any Shape at runtime ---
fn print_shape_dynamic(s: &dyn Shape) {
    println!("{s}: area={:.2}, perimeter={:.2}", s.area(), s.perimeter());
}

fn main() {
    let c = Circle { radius: 5.0 };
    let r = Rectangle { w: 4.0, h: 6.0 };

    print_shape_static(&c);   // monomorphized for Circle
    print_shape_static(&r);   // monomorphized for Rectangle

    print_shape_dynamic(&c);  // same function body, vtable dispatch
    print_shape_dynamic(&r);

    // Dynamic dispatch is REQUIRED when the type is not known at compile time:
    // storing mixed types in a Vec.
    let shapes: Vec<Box<dyn Shape>> = vec![
        Box::new(Circle { radius: 3.0 }),
        Box::new(Rectangle { w: 2.0, h: 5.0 }),
        Box::new(Circle { radius: 1.0 }),
    ];

    let total_area: f64 = shapes.iter().map(|s| s.area()).sum();
    println!("total area: {total_area:.2}");
}

Object Safety

Not every trait can be used as a trait object. A trait is object-safe only if the compiler can build a vtable for it. The rules are:

  • All methods must be callable through a pointer: they must not have a Self: Sized bound or return Self by value (since the size of Self is unknown through a trait object).
  • Methods must not have generic type parameters (the vtable would need an infinite number of entries).
  • The trait must not have associated functions without a receiver (i.e., "static methods" without &self are not dispatchable).

The Clone trait is not object-safe because its clone(&self) -> Self method returns Self by value. Iterator is object-safe because next(&mut self) -> Option<Self::Item> uses an associated type, not a generic parameter.

Demonstrating object safety rules: a non-object-safe trait, why it fails, and a fixed version.

// NOT object-safe: `clone` returns `Self` by value
// trait NotObjectSafe {
//     fn clone(&self) -> Self; // ERROR if used as dyn NotObjectSafe
// }

// NOT object-safe: generic method
// trait NotObjectSafe2 {
//     fn compare<T>(&self, other: T) -> bool; // ERROR
// }

// Object-safe: all methods take &self or &mut self, no generics, no Self return
trait Drawable {
    fn draw(&self);
    fn bounding_box(&self) -> (f64, f64, f64, f64); // (x, y, w, h)
}

// You CAN mark individual methods as non-dispatchable with `where Self: Sized`
// so the rest of the trait remains object-safe:
trait Cloneable {
    fn clone_box(&self) -> Box<dyn Cloneable>;

    // `where Self: Sized` excludes this method from the vtable; it is not
    // callable on a trait object, but the trait itself remains object-safe.
    fn type_name(&self) -> &'static str where Self: Sized {
        std::any::type_name::<Self>()
    }
}

struct Pixel { x: u32, y: u32 }
impl Drawable for Pixel {
    fn draw(&self) { println!("pixel at ({}, {})", self.x, self.y); }
    fn bounding_box(&self) -> (f64, f64, f64, f64) {
        (self.x as f64, self.y as f64, 1.0, 1.0)
    }
}

fn render(items: &[&dyn Drawable]) {
    for item in items { item.draw(); }
}

fn main() {
    let p1 = Pixel { x: 10, y: 20 };
    let p2 = Pixel { x: 30, y: 40 };
    render(&[&p1, &p2]);
}

Box<dyn Trait> — Heap-Allocated Trait Objects

Box<dyn Trait> is the most common form of owned trait object. The value is heap-allocated, the Box owns it, and when the Box is dropped the underlying value is dropped through the vtable's drop glue. This is the idiomatic way to store heterogeneous values in a collection, return a trait object from a function when the concrete type varies, or hold a field whose type is not known until runtime.

A plugin system that registers and runs heterogeneous plugins stored as Box<dyn Plugin>.

trait Plugin {
    fn name(&self) -> &str;
    fn run(&self, input: &str) -> String;
}

struct UppercasePlugin;
struct ReversePlugin;
struct RepeatPlugin { times: usize }

impl Plugin for UppercasePlugin {
    fn name(&self) -> &str { "uppercase" }
    fn run(&self, input: &str) -> String { input.to_uppercase() }
}
impl Plugin for ReversePlugin {
    fn name(&self) -> &str { "reverse" }
    fn run(&self, input: &str) -> String { input.chars().rev().collect() }
}
impl Plugin for RepeatPlugin {
    fn name(&self) -> &str { "repeat" }
    fn run(&self, input: &str) -> String { input.repeat(self.times) }
}

struct PluginRunner {
    plugins: Vec<Box<dyn Plugin>>,
}

impl PluginRunner {
    fn new() -> Self { PluginRunner { plugins: Vec::new() } }

    fn register(&mut self, p: Box<dyn Plugin>) {
        println!("Registered plugin: {}", p.name());
        self.plugins.push(p);
    }

    fn run_all(&self, input: &str) {
        for plugin in &self.plugins {
            println!("[{}] {}", plugin.name(), plugin.run(input));
        }
    }
}

fn main() {
    let mut runner = PluginRunner::new();
    runner.register(Box::new(UppercasePlugin));
    runner.register(Box::new(ReversePlugin));
    runner.register(Box::new(RepeatPlugin { times: 3 }));

    runner.run_all("hello");
    // [uppercase] HELLO
    // [reverse]   olleh
    // [repeat]    hellohellohello
}

Returning Trait Objects from Functions

When a function needs to return different concrete types depending on runtime conditions, impl Trait is not sufficient (all return paths must return the same type for impl Trait to work). The solution is Box<dyn Trait> as the return type. The function boxes each variant and returns the trait object. The caller sees a uniform type and can call trait methods without knowing the concrete implementation.

A factory function returning different concrete types through a Box<dyn Trait> return type, dispatched at runtime.

trait Formatter {
    fn format(&self, value: f64) -> String;
}

struct FixedFormatter { decimals: usize }
struct ScientificFormatter;
struct CurrencyFormatter { symbol: String }

impl Formatter for FixedFormatter {
    fn format(&self, v: f64) -> String { format!("{:.prec$}", v, prec = self.decimals) }
}
impl Formatter for ScientificFormatter {
    fn format(&self, v: f64) -> String { format!("{v:e}") }
}
impl Formatter for CurrencyFormatter {
    fn format(&self, v: f64) -> String { format!("{}{:.2}", self.symbol, v) }
}

fn make_formatter(style: &str) -> Box<dyn Formatter> {
    match style {
        "sci"      => Box::new(ScientificFormatter),
        "currency" => Box::new(CurrencyFormatter { symbol: "$".to_string() }),
        _          => Box::new(FixedFormatter { decimals: 2 }),
    }
}

fn main() {
    for style in ["fixed", "sci", "currency"] {
        let fmt = make_formatter(style);
        println!("{style}: {}", fmt.format(12345.6789));
    }
    // fixed:    12345.68
    // sci:      1.23456789e4
    // currency: $12345.68
}

Trait Objects with Lifetimes

A trait object has a lifetime: &'a dyn Trait or Box<dyn Trait + 'a>. When no lifetime is written, the compiler infers a default: &dyn Trait defaults to &'_ dyn Trait (some elided lifetime), and Box<dyn Trait> defaults to Box<dyn Trait + 'static>. The 'static bound on a boxed trait object means the concrete type must not hold any non-static references. This is common for storing callbacks, plugins, or event handlers that need to outlive the scope where they were created.

An event handler system using Box<dyn Fn() + Send + 'static> to store callbacks that are safe to send across threads and do not capture short-lived references.

use std::collections::HashMap;

struct EventBus {
    handlers: HashMap<String, Vec<Box<dyn Fn(&str) + Send + 'static>>>,
}

impl EventBus {
    fn new() -> Self { EventBus { handlers: HashMap::new() } }

    fn subscribe<F>(&mut self, event: &str, handler: F)
    where F: Fn(&str) + Send + 'static
    {
        self.handlers.entry(event.to_string()).or_default().push(Box::new(handler));
    }

    fn publish(&self, event: &str, payload: &str) {
        if let Some(handlers) = self.handlers.get(event) {
            for handler in handlers {
                handler(payload);
            }
        }
    }
}

fn main() {
    let mut bus = EventBus::new();

    bus.subscribe("user.login", |payload| println!("[audit] login: {payload}"));
    bus.subscribe("user.login", |payload| println!("[metrics] user logged in: {payload}"));
    bus.subscribe("user.logout", |payload| println!("[session] logout: {payload}"));

    bus.publish("user.login", "alice");
    // [audit]   login: alice
    // [metrics] user logged in: alice

    bus.publish("user.logout", "alice");
    // [session] logout: alice

    bus.publish("user.purchase", "item_42"); // no handlers registered
}

impl Trait vs dyn Trait — Decision Guide

Choosing between static and dynamic dispatch is a design decision with measurable trade-offs:

  • Use impl Trait (static) when: the concrete type is known at compile time, performance is critical (hot loop, tight numerical code), or you are writing a library and want callers to benefit from inlining.
  • Use dyn Trait (dynamic) when: you need to store multiple concrete types in the same collection, a function must return different concrete types from different branches, you are implementing a plugin or callback system, or compile-time binary size matters more than call overhead.
  • Prefer impl Trait as the default. Reach for dyn Trait only when static dispatch genuinely cannot express what you need.

A benchmark-style demonstration contrasting the two dispatch modes to make the performance difference tangible.

trait Compute {
    fn compute(&self, x: f64) -> f64;
}

struct Square;
struct Sine;

impl Compute for Square { fn compute(&self, x: f64) -> f64 { x * x } }
impl Compute for Sine   { fn compute(&self, x: f64) -> f64 { x.sin() } }

// Static: the compiler generates two versions, each fully inlined.
fn run_static(c: &impl Compute, data: &[f64]) -> f64 {
    data.iter().map(|&x| c.compute(x)).sum()
}

// Dynamic: one function, vtable call per element.
fn run_dynamic(c: &dyn Compute, data: &[f64]) -> f64 {
    data.iter().map(|&x| c.compute(x)).sum()
}

fn main() {
    let data: Vec<f64> = (0..1000).map(|i| i as f64 * 0.001).collect();

    let sq = Square;
    let si = Sine;

    // Static dispatch
    println!("square sum (static): {:.4}", run_static(&sq, &data));
    println!("sine sum (static):   {:.4}", run_static(&si, &data));

    // Dynamic dispatch
    println!("square sum (dynamic): {:.4}", run_dynamic(&sq, &data));
    println!("sine sum (dynamic):   {:.4}", run_dynamic(&si, &data));

    // The only scenario requiring dyn: a runtime-selected algorithm
    let algorithm: &dyn Compute = if std::env::var("USE_SINE").is_ok() { &si } else { &sq };
    println!("runtime selected: {:.4}", run_dynamic(algorithm, &data));
}
