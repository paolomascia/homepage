Interior Mutability: Cell, RefCell, Mutex & RwLock

Rust Interior Mutability Concurrency

Rust's borrow checker enforces a simple rule: you can have many immutable references or exactly one mutable reference, never both. Interior mutability is a design pattern that lets you mutate data through a shared (&T) reference by moving the borrow-checking rules from compile time to runtime. This is not a hack — it is a deliberate escape hatch for the cases where static analysis is too conservative, and every interior mutability type documents exactly which invariants it upholds and at what cost.

Cell<T> — Copy Types, No Borrow Overhead

Cell<T> provides interior mutability for types that implement Copy. It works by only ever allowing you to get the value (which copies it out) or set a new value — you never get a reference to the interior. Because there are no references, there are no aliasing violations: the borrow checker never needs to track anything. Cell is the fastest interior mutability primitive because it has zero runtime overhead: no counters, no locks, no unsafe at the call site.

Use Cell for small, Copy values that need to be mutated through shared references: flags, counters, cached computations in an otherwise immutable struct.

Cell in Practice

A struct with an immutable interface but a lazily-updated internal counter, implemented with Cell to avoid the cost of RefCell for a simple integer.

use std::cell::Cell;

struct HitCounter {
    name:   String,
    hits:   Cell<u64>,
    misses: Cell<u64>,
}

impl HitCounter {
    fn new(name: &str) -> Self {
        HitCounter { name: name.to_string(), hits: Cell::new(0), misses: Cell::new(0) }
    }

    // &self (not &mut self) -- exterior is immutable, interior is mutable.
    fn record_hit(&self)  { self.hits.set(self.hits.get() + 1); }
    fn record_miss(&self) { self.misses.set(self.misses.get() + 1); }

    fn hit_rate(&self) -> f64 {
        let h = self.hits.get() as f64;
        let m = self.misses.get() as f64;
        if h + m == 0.0 { 0.0 } else { h / (h + m) * 100.0 }
    }
}

// Cell for memoization: compute once, store the result
struct Fibonacci {
    n:    u64,
    memo: Cell<Option<u64>>,
}

impl Fibonacci {
    fn new(n: u64) -> Self { Fibonacci { n, memo: Cell::new(None) } }

    fn value(&self) -> u64 {
        if let Some(v) = self.memo.get() { return v; }
        let v = (0..self.n).fold((0u64, 1u64), |(a, b), _| (b, a + b)).0;
        self.memo.set(Some(v));
        v
    }
}

fn main() {
    let counter = HitCounter::new("cache");
    // Multiple shared references can coexist while mutating internals.
    let r1 = &counter;
    let r2 = &counter;
    r1.record_hit(); r1.record_hit(); r2.record_miss();
    println!("{}: hit rate = {:.1}%", counter.name, counter.hit_rate()); // 66.7%

    let fib = Fibonacci::new(40);
    println!("fib(40) = {}", fib.value()); // computed
    println!("fib(40) = {}", fib.value()); // from Cell cache
}

RefCell<T> — Dynamic Borrow Checking

RefCell<T> moves borrow checking to runtime. It maintains a counter of active borrows: each borrow() call increments a "readers" count and returns a Ref<T> guard; each borrow_mut() sets an exclusive "writer" flag and returns a RefMut<T> guard. If you try to borrow mutably when there is an active shared borrow (or vice versa), RefCell panics at runtime. The try_borrow and try_borrow_mut methods return Result instead of panicking.

RefCell is not Sync: it cannot be used from multiple threads. For multi-threaded interior mutability, use Mutex or RwLock. RefCell is the right choice when you know at design time that borrows will not overlap, but the borrow checker cannot verify this due to the structure of the code (for example, a graph algorithm that needs to both traverse and mutate, or a mock in tests).

RefCell Basics and the Borrow Rules at Runtime

Multiple shared borrows, exclusive mutable borrows, the panicking behaviour on borrow violations, and try_borrow for graceful error handling.

use std::cell::RefCell;

fn main() {
    let data = RefCell::new(vec![1, 2, 3]);

    // Multiple immutable borrows at the same time -- fine.
    {
        let r1 = data.borrow();
        let r2 = data.borrow();
        println!("r1={:?}, r2={:?}", *r1, *r2);
    } // r1 and r2 dropped here

    // Mutable borrow after all immutable borrows are released -- fine.
    {
        let mut w = data.borrow_mut();
        w.push(4);
        println!("after push: {:?}", *w);
    }

    // try_borrow: no panic, returns Err on violation
    let r = data.borrow();
    let result = data.try_borrow_mut(); // Err: immutable borrow active
    println!("try_borrow_mut while borrowed: {}", result.is_err()); // true
    drop(r);
    let result = data.try_borrow_mut(); // Ok: no active borrows
    println!("try_borrow_mut after drop: {}", result.is_ok()); // true

    // Interior mutation through a shared reference
    let shared = std::rc::Rc::new(RefCell::new(0i32));
    let a = std::rc::Rc::clone(&shared);
    let b = std::rc::Rc::clone(&shared);
    *a.borrow_mut() += 10;
    *b.borrow_mut() += 20;
    println!("shared value: {}", shared.borrow()); // 30
}

RefCell for Mock Objects in Tests

A classic use case: a test double that records calls through a &self interface using RefCell for interior mutability.

use std::cell::RefCell;

trait Messenger {
    fn send(&self, msg: &str);
}

// Real implementation
struct EmailMessenger;
impl Messenger for EmailMessenger {
    fn send(&self, msg: &str) { println!("[email] {msg}"); }
}

// Test double: records all sent messages
struct MockMessenger {
    messages: RefCell<Vec<String>>,
}

impl MockMessenger {
    fn new() -> Self { MockMessenger { messages: RefCell::new(Vec::new()) } }
    fn messages(&self) -> Vec<String> { self.messages.borrow().clone() }
}

impl Messenger for MockMessenger {
    // Trait requires &self, but we need to mutate messages -- RefCell enables this.
    fn send(&self, msg: &str) {
        self.messages.borrow_mut().push(msg.to_string());
    }
}

struct RateLimiter<'a, M: Messenger> {
    messenger: &'a M,
    count:     u32,
    limit:     u32,
}

impl<'a, M: Messenger> RateLimiter<'a, M> {
    fn new(messenger: &'a M, limit: u32) -> Self {
        RateLimiter { messenger, count: 0, limit }
    }

    fn request(&mut self) {
        self.count += 1;
        if self.count == self.limit {
            self.messenger.send("Warning: at 100% of quota");
        } else if self.count > self.limit {
            self.messenger.send("Error: quota exceeded");
        }
    }
}

fn main() {
    let mock = MockMessenger::new();
    let mut limiter = RateLimiter::new(&mock, 3);
    for _ in 0..4 { limiter.request(); }

    let msgs = mock.messages();
    println!("{msgs:?}");
    // ["Warning: at 100% of quota", "Error: quota exceeded"]
}

Mutex<T> — Thread-Safe Exclusive Access

Mutex<T> (mutual exclusion lock) provides thread-safe interior mutability. lock() blocks the calling thread until the lock is acquired, then returns a MutexGuard<T> that gives mutable access to the inner value. When the guard is dropped (at the end of its scope), the lock is released automatically. try_lock() is the non-blocking variant.

A critical detail: if a thread panics while holding a MutexGuard, the mutex enters a poisoned state. Subsequent calls to lock() return Err(PoisonError). You can recover the inner value via .into_inner() on the error, but it may be in an inconsistent state. Most production code calls .unwrap() on the lock result, which re-panics on a poisoned mutex — propagating the failure rather than silently proceeding with corrupted data.

Mutex Patterns

Scoped locking, minimising lock hold time, avoiding deadlocks, and a producer-consumer pattern with Mutex + Condvar.

use std::sync::{Arc, Mutex, Condvar};
use std::thread;

fn main() {
    // Basic: Arc<Mutex<T>> for shared mutable state
    let counter = Arc::new(Mutex::new(0u32));
    let mut handles = Vec::new();

    for _ in 0..10 {
        let c = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut val = c.lock().unwrap();
            *val += 1;
            // Lock released here when `val` (MutexGuard) is dropped
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("counter: {}", counter.lock().unwrap()); // 10

    // Minimise lock hold time: compute outside the lock
    let data = Arc::new(Mutex::new(Vec::<i32>::new()));
    let d = Arc::clone(&data);
    thread::spawn(move || {
        let expensive = (0..1000).sum::<i32>(); // compute WITHOUT holding the lock
        d.lock().unwrap().push(expensive);        // hold lock only for the push
    }).join().unwrap();

    // Condvar: block until a condition is met
    let pair = Arc::new((Mutex::new(false), Condvar::new()));
    let pair2 = Arc::clone(&pair);

    thread::spawn(move || {
        thread::sleep(std::time::Duration::from_millis(10));
        let (lock, cvar) = &*pair2;
        *lock.lock().unwrap() = true;
        cvar.notify_one(); // wake up the waiting thread
    });

    let (lock, cvar) = &*pair;
    let _guard = cvar.wait_while(lock.lock().unwrap(), |&mut ready| !ready).unwrap();
    println!("condition met");
}

RwLock<T> — Concurrent Reads, Exclusive Writes

RwLock<T> is a reader-writer lock. It allows any number of concurrent readers (read() → RwLockReadGuard) as long as there are no writers, and gives exclusive access to a single writer (write() → RwLockWriteGuard). This is the right choice when reads vastly outnumber writes and you want to maximise read concurrency without sacrificing correctness.

The trade-off vs Mutex: RwLock is more complex internally and can be slower than Mutex in write-heavy workloads due to the overhead of tracking reader counts. Benchmark before assuming RwLock is faster.

RwLock for a Read-Heavy Configuration Store

A global configuration object that many threads read concurrently but only an admin thread writes, using RwLock to maximise read throughput.

use std::sync::{Arc, RwLock};
use std::collections::HashMap;
use std::thread;

struct ConfigStore {
    data: RwLock<HashMap<String, String>>,
}

impl ConfigStore {
    fn new() -> Self { ConfigStore { data: RwLock::new(HashMap::new()) } }

    fn get(&self, key: &str) -> Option<String> {
        // Multiple threads can hold read locks simultaneously
        self.data.read().unwrap().get(key).cloned()
    }

    fn set(&self, key: &str, value: &str) {
        // Exclusive write lock: blocks until all readers release
        self.data.write().unwrap().insert(key.to_string(), value.to_string());
    }
}

fn main() {
    let store = Arc::new(ConfigStore::new());
    store.set("timeout", "30");
    store.set("max_connections", "100");

    // Spawn 8 reader threads
    let mut handles: Vec<_> = (0..8).map(|i| {
        let s = Arc::clone(&store);
        thread::spawn(move || {
            let v = s.get("timeout").unwrap_or_default();
            println!("reader {i}: timeout={v}");
        })
    }).collect();

    // Spawn 1 writer thread
    handles.push({
        let s = Arc::clone(&store);
        thread::spawn(move || {
            s.set("timeout", "60");
            println!("writer: updated timeout to 60");
        })
    });

    for h in handles { h.join().unwrap(); }
    println!("final timeout: {:?}", store.get("timeout"));

    // try_read / try_write: non-blocking, returns Err if lock cannot be acquired
    let lock = Arc::new(RwLock::new(0));
    let _r = lock.read().unwrap();
    let result = lock.try_write(); // Err: readers active
    println!("try_write with reader: {}", result.is_err()); // true
}

Choosing Interior Mutability

A decision guide for the four interior mutability primitives:

  • Cell<T> — single thread, T: Copy, no references to interior needed. Fastest: zero runtime cost.
  • RefCell<T> — single thread, T may not be Copy, dynamic borrow checking at runtime. Panics on violated borrow rules. Use try_borrow/try_borrow_mut if you need graceful error handling.
  • Mutex<T> — multiple threads, exclusive access. Blocks on contention. Correct for any access pattern (read or write).
  • RwLock<T> — multiple threads, many concurrent readers and infrequent writers. More overhead than Mutex; only beneficial when reads dominate.

Outside of these, OnceCell and OnceLock (for one-time initialization), AtomicUsize/AtomicBool (for lock-free counters and flags), and parking_lot crate (faster, feature-richer Mutex/RwLock) are common in production code.

AtomicUsize for a lock-free hit counter shared across threads.

use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let counter = Arc::new(AtomicUsize::new(0));
    let handles: Vec<_> = (0..10).map(|_| {
        let c = Arc::clone(&counter);
        thread::spawn(move || {
            // fetch_add: atomically increment and return previous value
            c.fetch_add(1, Ordering::Relaxed);
        })
    }).collect();
    for h in handles { h.join().unwrap(); }
    println!("atomic counter: {}", counter.load(Ordering::Relaxed)); // 10

    // OnceCell: initialize a value exactly once, then read it many times
    use std::sync::OnceLock;
    static CONFIG: OnceLock<String> = OnceLock::new();
    CONFIG.get_or_init(|| "production".to_string());
    println!("config: {}", CONFIG.get().unwrap());
}
