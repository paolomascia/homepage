Unsafe Rust: Raw Pointers, unsafe Blocks & FFI

Rust Unsafe FFI

Rust's safety guarantees are enforced by the compiler, but not every correct program can be expressed within those guarantees. unsafe is a conscious escape hatch that lets you step outside the checked subset when you have manually verified that the invariants the compiler would normally enforce still hold. The critical insight: unsafe does not disable safety — it transfers responsibility from the compiler to you.

What unsafe Enables

An unsafe block or function unlocks exactly five additional capabilities that are otherwise forbidden:

  1. Dereference a raw pointer (*const T or *mut T)
  2. Call an unsafe function or method
  3. Access or modify a mutable static variable
  4. Implement an unsafe trait
  5. Access fields of a union

Everything else remains checked. The borrow checker still operates inside unsafe blocks; lifetimes and types are still verified. Only the specific operations above are permitted beyond the normal rules. This containment is what makes unsafe sustainable: bugs can only be introduced through these five specific doors, not through arbitrary UB.

Raw Pointers

Raw pointers come in two flavours: *const T (immutable, like &T without lifetime tracking) and *mut T (mutable, like &mut T without borrow checking). You can create raw pointers from references in safe code; you need unsafe only to dereference them. Raw pointers are not automatically null-checked, not tracked by the borrow checker, and not automatically freed. They are exactly what C and C++ pointers are.

Raw Pointer Basics

Creating raw pointers from references and integers, null checks, dereferencing, and pointer arithmetic.

fn main() {
    let x = 42i32;

    // Create raw pointers from references -- safe
    let raw_const: *const i32 = &x as *const i32;
    let mut y = 10i32;
    let raw_mut: *mut i32 = &mut y as *mut i32;

    // Dereference requires unsafe
    unsafe {
        println!("*raw_const = {}", *raw_const); // 42
        *raw_mut += 5;
        println!("y = {}", *raw_mut); // 15
    }
    println!("y after unsafe: {y}"); // 15

    // Null pointer (NOT the same as None -- no compiler checking)
    let null_ptr: *const i32 = std::ptr::null();
    println!("is null: {}", null_ptr.is_null()); // true

    // Always check before dereferencing a possibly-null pointer
    if !null_ptr.is_null() {
        unsafe { println!("{}", *null_ptr); } // never reached
    }

    // Pointer arithmetic: advance by sizeof(T) bytes
    let arr = [10i32, 20, 30, 40, 50];
    let ptr: *const i32 = arr.as_ptr();
    unsafe {
        for i in 0..arr.len() {
            print!("{} ", *ptr.add(i)); // 10 20 30 40 50
        }
        println!();
    }

    // from_raw_parts: construct a slice from a raw pointer and length
    unsafe {
        let slice = std::slice::from_raw_parts(ptr, arr.len());
        println!("slice sum: {}", slice.iter().sum::<i32>()); // 150
    }
}

Safe Abstractions over Unsafe Code

The idiomatic approach is to encapsulate unsafe inside a safe abstraction: a module or type whose public API is entirely safe, but whose internal implementation uses unsafe where necessary. The unsafe code is small, auditable, and contained. Every Vec, Mutex, and channel in the standard library is built this way.

A split_at_mut reimplementation that produces two non-overlapping mutable slices from one, using raw pointers internally with a safe public interface.

/// Split `data` into two non-overlapping mutable slices at `mid`.
/// Safe precondition: `mid <= data.len()`.
/// SAFETY: the two raw pointers are derived from `data`, they point to
/// non-overlapping regions [0, mid) and [mid, len), and both are valid
/// for the lifetime of `data`.
fn split_at_mut_safe<T>(data: &mut [T], mid: usize) -> (&mut [T], &mut [T]) {
    assert!(mid <= data.len(), "mid out of bounds");
    let len = data.len();
    let ptr = data.as_mut_ptr();
    unsafe {
        (
            std::slice::from_raw_parts_mut(ptr, mid),
            std::slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}

fn main() {
    let mut v = vec![1, 2, 3, 4, 5, 6];
    let (left, right) = split_at_mut_safe(&mut v, 3);

    left.iter_mut().for_each(|x| *x *= 10);
    right.iter_mut().for_each(|x| *x += 100);

    println!("{:?}", v); // [10, 20, 30, 104, 105, 106]
}

unsafe Functions and Traits

Marking a function unsafe fn signals to callers that they must uphold certain invariants that the compiler cannot verify. The function's documentation should explicitly state these invariants as a # Safety section. An unsafe trait means that implementing it requires the implementor to manually uphold invariants: Send and Sync are the canonical examples — the compiler automatically derives them when it can verify the invariants, but you can opt in manually with unsafe impl when you have a type the compiler cannot reason about (e.g., one containing raw pointers).

An unsafe function with a documented safety contract, and an unsafe trait implementation for a raw-pointer-backed type.

use std::fmt;

/// A non-null, non-owning pointer to a T.
/// SAFETY: The caller must ensure the pointer remains valid for the
/// lifetime of any NonNullRef instance and is properly aligned.
struct NonNullRef<T> {
    ptr: *const T,
}

impl<T> NonNullRef<T> {
    /// # Safety
    /// `ptr` must be non-null, properly aligned, and valid for reads
    /// for as long as this NonNullRef exists.
    unsafe fn new(ptr: *const T) -> Self {
        assert!(!ptr.is_null());
        NonNullRef { ptr }
    }

    fn get(&self) -> &T {
        // SAFETY: constructor guarantee -- ptr is valid and aligned
        unsafe { &*self.ptr }
    }
}

// SAFETY: T: Send means the pointed-to data can be sent across threads.
// NonNullRef contains only a raw pointer, which is not Send by default;
// we are manually asserting that our use is correct.
unsafe impl<T: Send + Sync> Send for NonNullRef<T> {}
unsafe impl<T: Send + Sync> Sync for NonNullRef<T> {}

impl<T: fmt::Debug> fmt::Debug for NonNullRef<T> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "NonNullRef({:?})", self.get())
    }
}

fn main() {
    let value = 42i32;
    let r = unsafe { NonNullRef::new(&value) };
    println!("{r:?}"); // NonNullRef(42)
    println!("value: {}", r.get()); // 42

    // The reference behaves safely because the invariants are upheld
    let sum = *r.get() + 8;
    println!("sum: {sum}"); // 50
}

FFI — Calling C from Rust

Foreign Function Interface (FFI) lets Rust code call functions from C libraries and vice versa. On the Rust side, you declare the C function signature in an extern "C" block. Calling any extern function requires unsafe because Rust cannot verify that the C code upholds memory safety. The libc crate provides Rust bindings for the standard C library types and functions.

Type mapping between C and Rust requires care. The std::ffi and libc modules provide c_int, c_char, c_void, and other C-compatible types. Strings need special handling: C strings are null-terminated byte sequences, while Rust strings are UTF-8 fat pointers. CString and CStr bridge this gap.

Calling C Standard Library Functions

Calling strlen, abs, and qsort from C, with correct type conversions and null-termination handling.

use std::ffi::{CString, CStr, c_void};
use std::os::raw::{c_char, c_int};

extern "C" {
    fn strlen(s: *const c_char) -> usize;
    fn abs(n: c_int) -> c_int;
    fn qsort(
        base:  *mut c_void,
        nmemb: usize,
        size:  usize,
        compar: unsafe extern "C" fn(*const c_void, *const c_void) -> c_int,
    );
}

unsafe extern "C" fn compare_i32(a: *const c_void, b: *const c_void) -> c_int {
    let a = *(a as *const i32);
    let b = *(b as *const i32);
    a.cmp(&b) as c_int
}

fn main() {
    // strlen: C string length
    let s = CString::new("hello world").unwrap();
    let len = unsafe { strlen(s.as_ptr()) };
    println!("strlen: {len}"); // 11

    // abs: C absolute value
    let n: c_int = -42;
    println!("abs(-42) = {}", unsafe { abs(n) }); // 42

    // qsort: C quicksort
    let mut data: Vec<i32> = vec![5, 2, 8, 1, 9, 3, 7, 4, 6];
    unsafe {
        qsort(
            data.as_mut_ptr() as *mut c_void,
            data.len(),
            std::mem::size_of::<i32>(),
            compare_i32,
        );
    }
    println!("sorted: {data:?}"); // [1, 2, 3, 4, 5, 6, 7, 8, 9]

    // CStr: borrow a C string without allocation
    let c_hello = b"hello\0";
    let cstr = unsafe { CStr::from_bytes_with_nul_unchecked(c_hello) };
    println!("cstr: {:?}", cstr.to_str().unwrap()); // "hello"
}

Exporting Rust Functions to C

To make a Rust function callable from C, annotate it with #[no_mangle] (preserve the symbol name) and extern "C" (use the C calling convention). The function must only use C-compatible types at its boundary. This is how Rust libraries expose APIs to Python (via ctypes/cffi), C++, or other languages.

A small math library with Rust functions exported as a C-compatible shared library API.

use std::ffi::CStr;
use std::os::raw::c_char;

/// # Safety
/// `s` must be a valid, null-terminated C string.
#[no_mangle]
pub unsafe extern "C" fn rust_strlen(s: *const c_char) -> usize {
    if s.is_null() { return 0; }
    CStr::from_ptr(s).to_bytes().len()
}

#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}

#[no_mangle]
pub extern "C" fn rust_dot_product(
    a:   *const f64,
    b:   *const f64,
    len: usize,
) -> f64 {
    // SAFETY: caller guarantees a and b point to valid arrays of `len` f64s
    unsafe {
        let a = std::slice::from_raw_parts(a, len);
        let b = std::slice::from_raw_parts(b, len);
        a.iter().zip(b.iter()).map(|(x, y)| x * y).sum()
    }
}

// In Cargo.toml, set crate-type = ["cdylib"] to produce a .so/.dll
// Then from C:
//   extern int rust_add(int a, int b);
//   printf("%d\n", rust_add(3, 4)); // 7

fn main() {
    println!("rust_add(3, 4) = {}", rust_add(3, 4));
    let a = [1.0, 2.0, 3.0];
    let b = [4.0, 5.0, 6.0];
    println!("dot product = {}",
        rust_dot_product(a.as_ptr(), b.as_ptr(), a.len())); // 32.0
}

mem::transmute and mem::forget

std::mem::transmute<A, B>(val) reinterprets the bits of a value of type A as type B. It is the most dangerous function in the standard library: it bypasses all type safety and imposes no runtime checks. The types must have the same size; beyond that, anything goes. Use it only as a last resort when no safe conversion exists and you have a precise proof that the bit patterns are compatible.

std::mem::forget(val) takes ownership of a value and prevents its destructor from running. This is occasionally needed in FFI when ownership is transferred to C code that will free the memory. It is safe to call (hence no unsafe requirement), because the only consequence is a potential memory or resource leak — not memory unsafety.

Legitimate uses of transmute (f32 bit inspection) and forget (transferring ownership to C), with extensive commentary on why alternatives should always be tried first.

fn main() {
    // transmute: inspect f32 bits (use f32::to_bits() in real code!)
    let f: f32 = 1.0;
    let bits: u32 = unsafe { std::mem::transmute(f) };
    println!("1.0f32 bits: 0x{bits:08X}"); // 0x3F800000

    // to_bits() is the safe, idiomatic alternative:
    let bits2 = f.to_bits();
    assert_eq!(bits, bits2);

    // transmute lifetime extension -- extremely dangerous:
    // creates a reference with an incorrect lifetime
    // NEVER do this unless you have a very specific reason and proof
    // Example of what NOT to do:
    // let extended: &'static str = unsafe { std::mem::transmute("hello") };

    // forget: prevent the destructor of a value from running
    // Useful when transferring ownership to C or when a struct
    // manages a resource that should not be freed at this point.
    let v = vec![1, 2, 3];
    let ptr = v.as_ptr();
    let len = v.len();
    std::mem::forget(v); // v's destructor will NOT run; memory is not freed

    // Now the memory is "owned" by raw_vec -- must be freed manually!
    // In real FFI code, the C side would call free() on ptr.
    // Here we just reconstruct to avoid a leak in this example:
    let _recovered = unsafe { Vec::from_raw_parts(ptr as *mut i32, len, len) };
    // _recovered's destructor WILL free the memory

    // mem::size_of and align_of: useful for low-level layout work
    println!("size of i32:  {} bytes", std::mem::size_of::<i32>());
    println!("size of u64:  {} bytes", std::mem::size_of::<u64>());
    println!("size of &str: {} bytes", std::mem::size_of::<&str>());    // 16 (fat ptr)
    println!("align of f64: {} bytes", std::mem::align_of::<f64>());
}

When to Use unsafe — Practical Guidelines

  • Do not use unsafe to work around compiler errors. If the borrow checker rejects your code, that is almost always a signal that the code has a genuine ownership or aliasing problem. Fix the design, do not suppress the check.
  • Minimise the unsafe surface. Keep unsafe blocks as small as possible. Wrap unsafe operations in safe abstractions with documented invariants. Test the abstraction thoroughly.
  • Document the safety invariants. Every unsafe function and every unsafe block should have a # Safety comment explaining what preconditions must hold and why they are satisfied at this call site.
  • Use safe alternatives when they exist. f32::to_bits() instead of transmute. slice::split_at_mut instead of dual raw pointer arithmetic. CString instead of manual null termination.
  • Use tools. Miri (the Rust interpreter) detects undefined behaviour in unsafe code at test time. Sanitizers (AddressSanitizer, MemorySanitizer) catch memory errors at runtime.
