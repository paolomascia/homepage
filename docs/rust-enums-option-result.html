Enums, Option & Result in Rust

Rust Enums Option Result

Rust's type system is built on two fundamental abstractions that most languages handle poorly: the ability to say "this value is one of several distinct possibilities, each carrying its own data," and the ability to make the absence of a value or the failure of an operation explicit, first-class, and impossible to silently ignore. Enums, Option<T>, and Result<T, E> are the tools that deliver these guarantees — and the remarkable thing is that Option and Result are themselves just ordinary enums defined in the standard library, not language magic.

Enums in Rust

In C, an enum is syntactic sugar for a set of named integer constants. In Java, enums are slightly richer — they are class instances and can have methods — but each variant is still fundamentally the same type: it holds no per-variant data beyond what you manually add through fields on the class. Rust enums are something qualitatively different: they are Algebraic Data Types (ADTs), specifically sum types. A sum type says "a value of this type is exactly one of these named possibilities at any given moment, and each possibility can carry its own distinct payload."

This seemingly small difference has enormous practical consequences. In C, if you want a value that is either a circle with a radius or a rectangle with a width and height, you reach for a union combined with a tag field — an error-prone pattern because the compiler does not enforce that you only read the union member that corresponds to the tag. In Rust, you express this directly as an enum, and the compiler guarantees at every use site that you handle every variant. There is no way to accidentally read the rectangle's width when the value is actually a circle.

The reason Rust is designed this way is rooted in a philosophy that runs through the entire language: the type system should make incorrect states unrepresentable. If your domain has five distinct cases, you define an enum with five variants. The compiler then becomes your documentation, your test, and your runtime check simultaneously. Every match on that enum must be exhaustive — if you add a sixth variant later, the compiler will point you to every place in the codebase where you need to decide what to do with it.

Defining Basic Enums

A simple enum with unit variants (no data), matched exhaustively with match and stored in a variable.

// Unit variants: each variant holds no data, just a name
#[derive(Debug)]
enum Direction {
    North,
    South,
    East,
    West,
}

fn describe_direction(dir: &Direction) -> &'static str {
    match dir {
        Direction::North => "heading north",
        Direction::South => "heading south",
        Direction::East  => "heading east",
        Direction::West  => "heading west",
        // No wildcard needed — the compiler knows all variants are covered
    }
}

Enums with Data

The real power of Rust enums emerges when variants carry data. Each variant can hold a completely different payload: nothing at all (a unit variant), positional fields like a tuple (a tuple variant), or named fields like a struct (a struct variant). Think of it as a safe, typed union: the compiler tracks which variant is active and only allows you to access the data that actually exists for that variant. In C, a union with a tag field relies entirely on programmer discipline; in Rust, the type system enforces it unconditionally.

This is what makes enums the right tool for modeling domain concepts that are "one of several possibilities, each with its own shape." A network packet, a UI event, a configuration source, a parsed AST node — all of these fit naturally into an enum with data-carrying variants.

Three variant kinds demonstrated in one enum: a unit variant, a tuple variant (positional fields), and a struct variant (named fields).

#[derive(Debug)]
enum Shape {
    // Unit variant: no data needed
    Point,
    // Tuple variant: positional data — the radius
    Circle(f64),
    // Another tuple variant: two positional fields — width and height
    Rectangle(f64, f64),
    // Struct variant: named fields — clearer for three or more values
    Triangle { base: f64, height: f64 },
}

fn main() {
    let shapes = vec![
        Shape::Point,
        Shape::Circle(3.0),
        Shape::Rectangle(4.0, 5.0),
        Shape::Triangle { base: 6.0, height: 8.0 },
    ];

    for shape in &shapes {
        println!("{shape:?}");
    }
    // Point
    // Circle(3.0)
    // Rectangle(4.0, 5.0)
    // Triangle { base: 6.0, height: 8.0 }
}

Methods on Enums

Adding an impl block to the enum gives it methods; match self destructures each variant and binds its fields.

#[derive(Debug)]
enum Shape {
    Point,
    Circle(f64),
    Rectangle(f64, f64),
    Triangle { base: f64, height: f64 },
}

impl Shape {
    fn area(&self) -> f64 {
        match self {
            Shape::Point                          => 0.0,
            Shape::Circle(r)                      => std::f64::consts::PI * r * r,
            Shape::Rectangle(w, h)                => w * h,
            Shape::Triangle { base: b, height: h} => 0.5 * b * h,
        }
    }

    fn describe(&self) -> String {
        match self {
            Shape::Point           => "a point (no area)".to_string(),
            Shape::Circle(r)       => format!("a circle with radius {r:.2}"),
            Shape::Rectangle(w, h) => format!("a {w:.2}x{h:.2} rectangle"),
            Shape::Triangle { base, height } =>
                format!("a triangle with base {base:.2} and height {height:.2}"),
        }
    }
}

fn main() {
    let shapes = vec![
        Shape::Point,
        Shape::Circle(3.0),
        Shape::Rectangle(4.0, 5.0),
        Shape::Triangle { base: 6.0, height: 8.0 },
    ];

    for s in &shapes {
        println!("{}: area = {:.4}", s.describe(), s.area());
    }
}

The #[derive] Attributes on Enums

Deriving Debug, Clone, and PartialEq on an enum — enabling {:?} formatting, .clone(), and == comparison without writing any implementation code.

// The compiler generates Debug, Clone, and PartialEq implementations automatically
#[derive(Debug, Clone, PartialEq)]
enum Color {
    Red,
    Green,
    Blue,
    Custom(u8, u8, u8),
}

fn main() {
    let c1 = Color::Red;
    let c2 = c1.clone(); // Clone: make an independent copy

    // Debug: print with {:?}
    println!("{c1:?}");  // Red
    println!("{c2:?}");  // Red

    // PartialEq: compare with ==
    println!("{}", c1 == c2);                                    // true
    println!("{}", Color::Blue == Color::Green);                 // false
    println!("{}", Color::Custom(10, 20, 30) == Color::Custom(10, 20, 30)); // true
    println!("{}", Color::Custom(10, 20, 30) == Color::Custom(10, 20, 99)); // false

    let c3 = Color::Custom(255, 128, 0);
    println!("{c3:?}"); // Custom(255, 128, 0)
}

Representing State Machines

One of the most compelling uses of enums is modeling state machines. A state machine is a system that can be in exactly one of a finite set of states at any time, and that transitions between states according to defined rules. Enums are a perfect fit: the enum type represents the machine, each variant represents one state, and the compiler ensures you never accidentally treat one state as another or forget to handle a state in your transition logic.

This is something that is genuinely hard to do safely in languages without sum types. In Java or Python you might represent state as a string or an integer, but then there is nothing stopping you from putting an invalid value in the field, or from forgetting to handle a state in a long if-else chain. In Rust, if you add a new state to the enum, every match in the codebase that does not include a wildcard arm becomes a compile error, immediately pointing you to every piece of logic that needs to be updated. The compiler becomes a collaborator in maintaining the invariants of your state machine.

A traffic light modeled as an enum: exhaustive match for transitions and durations, iterated in a loop.

#[derive(Debug, Clone, PartialEq)]
enum TrafficLight {
    Red,
    Yellow,
    Green,
}

impl TrafficLight {
    // Returns the next state in the cycle
    fn next(&self) -> TrafficLight {
        match self {
            TrafficLight::Red    => TrafficLight::Green,
            TrafficLight::Green  => TrafficLight::Yellow,
            TrafficLight::Yellow => TrafficLight::Red,
        }
    }

    // Each state has a defined duration — the compiler ensures all states are covered
    fn duration_secs(&self) -> u32 {
        match self {
            TrafficLight::Red    => 30,
            TrafficLight::Green  => 25,
            TrafficLight::Yellow => 5,
        }
    }
}

fn main() {
    let mut light = TrafficLight::Red;

    // Simulate six transitions
    for _ in 0..6 {
        println!("{:?} — hold for {} seconds", light, light.duration_secs());
        light = light.next();
    }
    // Red — hold for 30 seconds
    // Green — hold for 25 seconds
    // Yellow — hold for 5 seconds
    // Red — hold for 30 seconds
    // ...
}

Option<T> — Null Safety

In 1965, Tony Hoare introduced the null reference into ALGOL W. He later called it his "billion dollar mistake" — an understatement by several orders of magnitude. Null references have caused an incalculable number of runtime crashes, security vulnerabilities, and debugging hours across virtually every language that adopted them. The problem is not that the concept of "absence" is wrong; it is that representing absence as a special value of every pointer type means the compiler cannot distinguish "this will always have a value" from "this might not have a value." Every pointer dereference is potentially unsafe, and there is no way to verify correctness without running the code.

Rust eliminates null entirely. There is no null, no nil, no undefined. The concept of "a value that might be absent" is represented instead by the type Option<T>, which is defined in the standard library as an ordinary enum. Because it is a distinct type, the compiler can enforce that you handle both the present and absent cases before you can access the inner value. You cannot accidentally dereference a "null" because the type system does not allow you to use the inner value without first checking which variant you have.

The definition of Option is beautifully simple:

// From the Rust standard library — just a regular enum
// T is a generic type parameter: Option works for any type
enum Option<T> {
    Some(T),  // the value is present, and here it is
    None,     // the value is absent
}

// The compiler knows about Option so you don't need to qualify it:
let present: Option<i32> = Some(42);
let absent: Option<i32>  = None;

// You cannot use the i32 inside present without handling None:
// let n = present + 1; // compile error — Option<i32> is not i32

Basic Option Usage

A function returning Option<i32> showing three idiomatic ways to consume it: exhaustive match, if let, and unwrap_or.

fn find_first_even(nums: &[i32]) -> Option<i32> {
    for &n in nums {
        if n % 2 == 0 {
            return Some(n); // found — wrap in Some
        }
    }
    None // not found — return None
}

fn main() {
    let numbers = vec![1, 3, 7, 4, 9, 2];

    // 1. Exhaustive match: the compiler forces handling of both arms
    match find_first_even(&numbers) {
        Some(n) => println!("found even: {n}"),
        None    => println!("no even numbers"),
    }

    // 2. if let: when you only care about the Some case
    if let Some(n) = find_first_even(&numbers) {
        println!("if let found: {n}");
    }

    // 3. unwrap_or: provide a fallback value if None
    let result = find_first_even(&[1, 3, 5]).unwrap_or(-1);
    println!("unwrap_or result: {result}"); // -1

    // 4. unwrap_or_else: lazy fallback — closure only called if None
    let result2 = find_first_even(&[1, 3, 5]).unwrap_or_else(|| {
        println!("computing fallback...");
        0
    });
    println!("unwrap_or_else result: {result2}"); // 0
}

Option Methods — a Rich API

Option<T> ships with a large set of methods that let you work with optional values in a fluent, composable style. Understanding these methods is important because they appear everywhere in idiomatic Rust code, and each one expresses a precise intent that would otherwise require several lines of match.

is_some() / is_none() — check which variant you have without consuming or unwrapping the value. Useful for conditional logic where you want to branch on presence without binding the inner value.

Checking whether an Option is populated, without extracting its contents.

let maybe: Option<i32> = Some(5);
println!("{}", maybe.is_some()); // true
println!("{}", maybe.is_none()); // false

let empty: Option<i32> = None;
println!("{}", empty.is_some()); // false
println!("{}", empty.is_none()); // true

unwrap() / expect() — extract the inner value, panicking if the option is None. unwrap() gives a generic panic message; expect("message") lets you provide a descriptive message that explains what invariant was violated. Always prefer expect in production code: a good message turns a mystery panic into a meaningful diagnosis.

Extracting values with unwrap and expect — and why expect is always preferable.

let val: Option<&str> = Some("hello");
println!("{}", val.unwrap()); // "hello" — panics if None with generic message

// expect gives a precise message when the invariant fails
let config_value: Option<String> = Some("production".to_string());
let env = config_value.expect("APP_ENV must be set before calling this function");
println!("{env}");

// None.unwrap()  => thread 'main' panicked at 'called `Option::unwrap()` on a `None` value'
// None.expect("context") => thread 'main' panicked at 'context'

unwrap_or(default) / unwrap_or_else(|| ...) — provide a fallback instead of panicking. unwrap_or eagerly evaluates the default (suitable for cheap values); unwrap_or_else accepts a closure that is only called when the option is None (suitable for expensive computations or side effects).

Eager vs lazy default values: use unwrap_or for cheap literals, unwrap_or_else for closures that do work.

// unwrap_or: default evaluated immediately (fine for literals)
let port: Option<u16> = None;
let actual_port = port.unwrap_or(8080);
println!("{actual_port}"); // 8080

// unwrap_or_else: closure only runs on None — important when default is expensive
let name: Option<String> = None;
let username = name.unwrap_or_else(|| {
    // Only called when name is None
    std::env::var("USER").unwrap_or_else(|_| "anonymous".to_string())
});
println!("{username}");

map(|x| ...) — transform the inner value of a Some by applying a function to it. If the option is None, map short-circuits and returns None without calling the function. This is the "functor" operation: it lets you work on the value inside the box without unpacking and repacking it manually.

Transforming the contents of a Some without touching None — the option equivalent of a safe transformation.

let maybe_num: Option<i32> = Some(4);
let doubled = maybe_num.map(|n| n * 2);
println!("{doubled:?}"); // Some(8)

let nothing: Option<i32> = None;
let still_nothing = nothing.map(|n| n * 2);
println!("{still_nothing:?}"); // None — closure never called

// map can change the inner type
let maybe_str: Option<i32> = Some(42);
let as_string: Option<String> = maybe_str.map(|n| n.to_string());
println!("{as_string:?}"); // Some("42")

and_then(|x| ...) — chain operations that themselves return Option. Also known as flatMap in other languages. Where map would produce Option<Option<T>>, and_then flattens the result to Option<T>. This is the fundamental tool for composing a series of fallible operations where any step might produce None.

Chaining two Option-returning operations — if either step produces None, the whole chain short-circuits.

fn parse_positive(s: &str) -> Option<u32> {
    s.trim().parse::<i32>().ok()
        .and_then(|n| if n > 0 { Some(n as u32) } else { None })
}

println!("{:?}", parse_positive("42"));   // Some(42)
println!("{:?}", parse_positive("-3"));   // None — negative
println!("{:?}", parse_positive("abc"));  // None — parse fails

or(other) and filter(|x| ...) — or provides a fallback Option when the first is None; filter converts a Some to None if the inner value does not satisfy a predicate. Together they let you express "use this, or try that" and "only proceed if the value meets this condition."

Fallback chaining with or and conditional filtering with filter.

let primary: Option<i32> = None;
let fallback: Option<i32> = Some(99);
println!("{:?}", primary.or(fallback)); // Some(99)

let val: Option<i32> = Some(10);
let filtered = val.filter(|&n| n > 5);
println!("{:?}", filtered); // Some(10)

let val2: Option<i32> = Some(3);
let filtered2 = val2.filter(|&n| n > 5);
println!("{:?}", filtered2); // None — 3 is not > 5

ok_or(err) — convert an Option<T> into a Result<T, E> by supplying an error value for the None case. This is the bridge from the "absence" world to the "error" world, and it is discussed further in the Combining section below.

Converting an Option to a Result by naming what the absence means as an error.

let maybe: Option<i32> = Some(7);
let result: Result<i32, &str> = maybe.ok_or("value was missing");
println!("{result:?}"); // Ok(7)

let nothing: Option<i32> = None;
let err_result: Result<i32, &str> = nothing.ok_or("value was missing");
println!("{err_result:?}"); // Err("value was missing")

Here is a realistic pipeline that chains several of these methods: parse a string as a number, keep it only if it is positive, then halve it.

A fluent Option pipeline combining and_then, filter, and map — each step can independently produce None, short-circuiting the rest.

fn process(input: &str) -> Option<f64> {
    input
        .trim()
        .parse::<f64>()  // returns Result — .ok() converts to Option
        .ok()
        .filter(|&n| n > 0.0)      // keep only positive values
        .map(|n| n / 2.0)          // halve it
}

fn main() {
    println!("{:?}", process("  8.0  ")); // Some(4.0)
    println!("{:?}", process("-3.0"));    // None — filtered out
    println!("{:?}", process("hello"));   // None — parse failed
    println!("{:?}", process("0.0"));     // None — not > 0
}

Nested Options and Flattening

Dealing with Option<Option<T>> using flatten(), and navigating a 2D grid safely with and_then.

// Nested Option arises when you map an Option-returning function over an Option
let nested: Option<Option<i32>> = Some(Some(42));
let flat: Option<i32> = nested.flatten();
println!("{flat:?}"); // Some(42)

let nested2: Option<Option<i32>> = Some(None);
println!("{:?}", nested2.flatten()); // None

let nested3: Option<Option<i32>> = None;
println!("{:?}", nested3.flatten()); // None

// Real-world use: safely index a 2D grid without panicking on out-of-bounds
let grid = vec![
    vec![1, 2, 3],
    vec![4, 5, 6],
    vec![7, 8, 9],
];

fn get_cell(grid: &Vec<Vec<i32>>, row: usize, col: usize) -> Option<i32> {
    // get() returns Option<&[i32]> — and_then chains to get the column
    grid.get(row).and_then(|r| r.get(col).copied())
}

println!("{:?}", get_cell(&grid, 1, 2)); // Some(6)
println!("{:?}", get_cell(&grid, 5, 0)); // None — row out of bounds
println!("{:?}", get_cell(&grid, 0, 9)); // None — col out of bounds

Result<T, E> — Recoverable Errors

Rust's approach to error handling is grounded in a simple idea: errors are values. A function that can fail says so in its return type. The caller receives either a success value or an error value, and the compiler ensures the caller acknowledges both possibilities. You cannot accidentally ignore an error the way you can in C (where a function might return -1 on failure and many callers forget to check), and you do not pay the runtime overhead and non-local control flow of exceptions.

Compare with Java's approach: checked exceptions force you to handle or declare errors, but the mechanism is separate from the return type — it is threaded through a parallel channel that the type system treats differently. Unchecked exceptions (subclasses of RuntimeException) can be thrown anywhere and caught nowhere, making it impossible to reason about which code paths can fail. Rust unifies everything: if a function can fail, the failure is visible in the type signature, handled with the same tools as success, and impossible to accidentally discard (the compiler emits a warning if you ignore a Result).

Like Option, Result is an ordinary enum in the standard library:

// From the Rust standard library
// T is the success type, E is the error type
enum Result<T, E> {
    Ok(T),   // the operation succeeded, here is the value
    Err(E),  // the operation failed, here is the error
}

// A function that can fail declares Result in its signature
// The caller is forced to handle both outcomes
fn might_fail(input: &str) -> Result<i32, String> {
    if input.is_empty() {
        Err("input was empty".to_string())
    } else {
        Ok(input.len() as i32)
    }
}

Basic Result Usage

A division function returning Result, consumed with an exhaustive match that handles the error case explicitly.

fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 {
        Err(format!("cannot divide {a} by zero"))
    } else {
        Ok(a / b)
    }
}

fn main() {
    // Exhaustive match — the compiler requires both Ok and Err arms
    match divide(10.0, 2.0) {
        Ok(result)  => println!("10 / 2 = {result}"),
        Err(msg)    => println!("error: {msg}"),
    }

    match divide(10.0, 0.0) {
        Ok(result)  => println!("result: {result}"),
        Err(msg)    => println!("error: {msg}"), // "cannot divide 10 by zero"
    }
}

Result Methods

Result<T, E> mirrors much of Option<T>'s API, with additional methods for transforming the error side. This symmetry is deliberate: once you understand Option's combinators, Result's feel immediately familiar.

is_ok() / is_err() — inspect which variant you have without consuming the value. Useful for assertions in tests or for routing logic that does not need the inner value.

Testing whether a Result represents success or failure without extracting either value.

let good: Result<i32, &str> = Ok(42);
let bad:  Result<i32, &str> = Err("oops");

println!("{} {}", good.is_ok(), good.is_err()); // true false
println!("{} {}", bad.is_ok(),  bad.is_err());  // false true

unwrap() / expect("message") — extract the Ok value, panicking on Err. As with Option, always prefer expect with a message that describes the invariant being asserted. Reserve these for cases where you have already ensured success through prior logic, or in quick prototypes and tests.

Using expect to communicate intent: the message tells future readers (and future you) what went wrong and why it was unexpected.

let config: Result<String, std::io::Error> =
    std::fs::read_to_string("config.toml");

// Prefer expect over unwrap: the message becomes the panic message on failure
let content = config.expect("config.toml must exist and be readable at startup");

// In tests, unwrap is fine because test failures should panic
// let parsed: i32 = "42".parse().unwrap();

unwrap_or / unwrap_or_else — return the Ok value or a fallback if Err, without panicking. The same eager vs lazy distinction as Option.

Providing a safe fallback value when an operation fails, avoiding panics in production code paths.

let result: Result<i32, &str> = Err("bad input");
println!("{}", result.unwrap_or(0));    // 0

let result2: Result<i32, String> = Err("bad".to_string());
let val = result2.unwrap_or_else(|e| {
    println!("recovering from error: {e}");
    -1
});
println!("{val}"); // -1

map(|x| ...) — transform the Ok value, leaving Err untouched. Identical in spirit to Option::map: apply a function to the success path without having to unwrap and rewrap.

Transforming the success value of a Result while leaving any error unchanged.

let ok: Result<i32, &str> = Ok(5);
let mapped = ok.map(|n| n * 10);
println!("{mapped:?}"); // Ok(50)

let err: Result<i32, &str> = Err("failed");
let mapped_err = err.map(|n| n * 10);
println!("{mapped_err:?}"); // Err("failed") — map not called

map_err(|e| ...) — transform the Err value, leaving Ok untouched. This is essential for converting between error types — a common need when combining results from different libraries that use different error types.

Converting an error type with map_err — the key tool for making disparate error types compatible.

// parse() returns Result<i32, ParseIntError>; we want Result<i32, String>
let result: Result<i32, String> =
    "42".parse::<i32>()
        .map_err(|e| format!("parse failed: {e}"));
println!("{result:?}"); // Ok(42)

let bad: Result<i32, String> =
    "xyz".parse::<i32>()
        .map_err(|e| format!("parse failed: {e}"));
println!("{bad:?}"); // Err("parse failed: invalid digit found in string")

and_then(|x| ...) — chain fallible operations. The closure receives the Ok value and must itself return a Result. If the first operation is Err, the chain short-circuits. This is the fundamental tool for sequencing operations where any step can fail.

Chaining two fallible operations: if either step fails, the whole chain returns that Err.

fn parse_number(s: &str) -> Result<i32, String> {
    s.trim().parse::<i32>().map_err(|e| format!("parse error: {e}"))
}

fn double_if_positive(n: i32) -> Result<i32, String> {
    if n > 0 {
        Ok(n * 2)
    } else {
        Err(format!("{n} is not positive"))
    }
}

let pipeline = parse_number("5").and_then(double_if_positive);
println!("{pipeline:?}"); // Ok(10)

let pipeline2 = parse_number("-3").and_then(double_if_positive);
println!("{pipeline2:?}"); // Err("-3 is not positive")

let pipeline3 = parse_number("abc").and_then(double_if_positive);
println!("{pipeline3:?}"); // Err("parse error: ...")

ok() — convert a Result<T, E> to Option<T>, discarding the error information. Useful when you care about presence or absence but not about the reason for failure.

Dropping the error and converting to Option when the distinction between failure reasons does not matter.

let ok_result: Result<i32, &str> = Ok(7);
println!("{:?}", ok_result.ok()); // Some(7)

let err_result: Result<i32, &str> = Err("failed");
println!("{:?}", err_result.ok()); // None — error info discarded

Here is a complete chained pipeline using map_err and and_then together:

A realistic pipeline: parse a string, validate the value, and double it — each step using the appropriate combinator.

fn parse_and_double(input: &str) -> Result<i32, String> {
    input
        .trim()
        .parse::<i32>()
        .map_err(|e| format!("could not parse '{input}': {e}"))
        .and_then(|n| {
            if n >= 0 {
                Ok(n * 2)
            } else {
                Err(format!("expected non-negative number, got {n}"))
            }
        })
}

fn main() {
    println!("{:?}", parse_and_double(" 21 "));  // Ok(42)
    println!("{:?}", parse_and_double("-5"));     // Err("expected non-negative number, got -5")
    println!("{:?}", parse_and_double("hello"));  // Err("could not parse 'hello': ...")
}

The ? Operator with Result

The ? operator is syntactic sugar for a very common pattern: if the value is Ok(x), unwrap it to x and continue; if it is Err(e), convert e using From::from (which handles error type conversion automatically when a From implementation exists) and return early from the current function. This makes error-propagation code dramatically more readable: instead of a ladder of match expressions, each fallible call gets a simple ? suffix. The function must itself return a Result (or Option) for ? to be valid.

The From::from conversion that ? performs automatically is why it is idiomatic to define a custom error enum and implement From<SpecificError> for it: you can then use ? across function calls that return different error types, and the conversions happen at each ? site without ceremony.

The ? operator in use: clean early-return propagation vs the equivalent verbose match desugaring.

use std::num::ParseIntError;

// With ?: each ? either unwraps Ok or returns Err from the function
fn parse_and_add(a: &str, b: &str) -> Result<i32, ParseIntError> {
    let x = a.trim().parse::<i32>()?;  // returns early with Err if parse fails
    let y = b.trim().parse::<i32>()?;  // same
    Ok(x + y)
}

// What the compiler desugars ? into (approximately):
fn parse_and_add_verbose(a: &str, b: &str) -> Result<i32, ParseIntError> {
    let x = match a.trim().parse::<i32>() {
        Ok(val) => val,
        Err(e)  => return Err(From::from(e)),
    };
    let y = match b.trim().parse::<i32>() {
        Ok(val) => val,
        Err(e)  => return Err(From::from(e)),
    };
    Ok(x + y)
}

fn main() {
    println!("{:?}", parse_and_add("10", "32"));    // Ok(42)
    println!("{:?}", parse_and_add("10", "abc"));   // Err(ParseIntError { ... })
    println!("{:?}", parse_and_add_verbose("3", "4")); // Ok(7)
}

Collecting Results: Vec of Results

A common pattern is to have an iterator of operations that each might fail, and you want either all the results (if every operation succeeded) or the first error (if any failed). Rust's collect() handles this transparently: if you collect into Result<Vec<T>, E>, it iterates through the results and returns Ok(Vec<T>) if all were Ok, or the first Err it encounters. This is far more ergonomic than manually looping and accumulating errors.

Collecting an iterator of fallible operations into either a complete Vec or the first error.

fn main() {
    // All succeed: collect into Result<Vec<i32>, _>
    let strings_ok = vec!["1", "2", "3", "4"];
    let result: Result<Vec<i32>, _> = strings_ok
        .iter()
        .map(|s| s.parse::<i32>())
        .collect();
    println!("{result:?}"); // Ok([1, 2, 3, 4])

    // One fails: collect returns the first Err
    let strings_bad = vec!["1", "two", "3"];
    let result2: Result<Vec<i32>, _> = strings_bad
        .iter()
        .map(|s| s.parse::<i32>())
        .collect();
    println!("{result2:?}"); // Err(ParseIntError { ... })
}

// Real-world example: parse a CSV row of floats
fn parse_csv_row(row: &str) -> Result<Vec<f64>, String> {
    row.split(',')
        .map(|field| {
            field.trim()
                 .parse::<f64>()
                 .map_err(|e| format!("could not parse '{}': {}", field.trim(), e))
        })
        .collect()  // collect stops at first Err
}

fn main_csv() {
    println!("{:?}", parse_csv_row("1.5, 2.0, 3.14")); // Ok([1.5, 2.0, 3.14])
    println!("{:?}", parse_csv_row("1.5, bad, 3.14")); // Err("could not parse 'bad': ...")
}

Combining Option and Result

Real programs constantly move between the "might be absent" world of Option and the "might have failed" world of Result. The standard library provides clean bridges in both directions. Option::ok_or(err) and Option::ok_or_else(|| err) convert a None into a named error, promoting absence into a failure with an explanation. Result::ok() goes the other way, converting an error into absence when the reason for failure does not matter to the caller. These conversions let you use ? on options inside functions that return Result, and use if let Some on results inside functions that return Option.

Converting between Option and Result in both directions, and using both with ? in a composed function.

use std::collections::HashMap;

// Option::ok_or_else: give the None case a specific error message
fn lookup_age(db: &HashMap<&str, i32>, name: &str) -> Result<i32, String> {
    db.get(name)
        .copied()
        .ok_or_else(|| format!("no entry for '{name}' in database"))
}

// Result::ok(): drop the error, treat failure as absence
fn is_adult(db: &HashMap<&str, i32>, name: &str) -> Option<bool> {
    lookup_age(db, name)
        .ok()           // Err becomes None, Ok(age) becomes Some(age)
        .map(|age| age >= 18)
}

// Use ? with Result in a function that calls both
fn check_entry(db: &HashMap<&str, i32>, name: &str) -> Result<String, String> {
    let age = lookup_age(db, name)?; // ? works because lookup_age returns Result
    if age >= 18 {
        Ok(format!("{name} is an adult (age {age})"))
    } else {
        Ok(format!("{name} is a minor (age {age})"))
    }
}

fn main() {
    let mut db = HashMap::new();
    db.insert("alice", 30);
    db.insert("bob", 15);

    println!("{:?}", lookup_age(&db, "alice")); // Ok(30)
    println!("{:?}", lookup_age(&db, "carol")); // Err("no entry for 'carol'...")

    println!("{:?}", is_adult(&db, "alice")); // Some(true)
    println!("{:?}", is_adult(&db, "bob"));   // Some(false)
    println!("{:?}", is_adult(&db, "carol")); // None

    println!("{:?}", check_entry(&db, "alice")); // Ok("alice is an adult (age 30)")
    println!("{:?}", check_entry(&db, "carol")); // Err("no entry for 'carol'...")
}

When to Use Enums vs Structs

Structs and enums complement each other as the two fundamental ways to compose data in Rust, corresponding to the two fundamental operations in type theory. A struct is a product type: it holds a value for field A and a value for field B and a value for field C, all simultaneously. Every field exists for every instance. An enum is a sum type: it holds a value that is variant A or variant B or variant C, with only one active at a time.

The practical rule is simple. If you are describing a thing that has multiple properties that all apply at once, use a struct. A User has a name and an email and a creation date. If you are describing a thing that can be in one of several distinct states, use an enum. An AuthState is either anonymous or logged in with a user or locked out with a reason — not all three simultaneously.

The compiler's exhaustiveness checking makes enums especially safe to evolve. Adding a new field to a struct is usually backward compatible (though it breaks struct literal initialization). Adding a new variant to an enum is immediately flagged everywhere in the codebase where that enum is matched without a wildcard arm — the compiler guides you to every decision point that needs to be updated. This is invaluable in large codebases: the type system enforces that your logic is consistent with your data model.

In practice, structs and enums are often combined: enum variants frequently carry struct-like data, and structs frequently contain fields whose types are enums. Option<T> itself is an enum that appears as a field in countless structs. The two abstractions are not alternatives but complements, each expressing a different kind of relationship between values.
