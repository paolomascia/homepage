Error Handling with thiserror & anyhow in Rust

Rust Error Handling thiserror anyhow

Rust has no exceptions. Instead, functions that can fail return Result<T, E> — a value you must inspect before you can use the success path. This makes failures visible, explicit, and impossible to silently ignore. The standard library provides the foundation; the thiserror and anyhow crates build on top of it to cover the two dominant error-handling patterns: defining rich, typed errors for libraries, and propagating heterogeneous errors ergonomically in applications.

The Standard Library Foundation

Before reaching for external crates, it is worth understanding exactly what the standard library provides. The std::error::Error trait is the universal marker for error types. It has a blanket implementation requirement that the type also implement Debug and Display — the first for machine-readable output (logs, debug builds), the second for human-readable messages. It optionally provides a source() method to expose the underlying cause, enabling error chain traversal.

The ? operator is the key piece of syntactic sugar. When applied to a Result<T, E> inside a function that returns Result<T2, E2>, it either unwraps the Ok value or returns early with the error, converting it from E to E2 using the From trait. This means that as long as you implement From<E> for E2 (or the crates you use do it for you), the ? operator becomes a one-character propagator that handles the entire unwrap-or-return-error-with-conversion dance.

Manual Error Types (stdlib only)

To understand what thiserror is saving you from, it helps to see what implementing an error type manually looks like. You need to implement Debug (usually derived), Display (manually formatted), and std::error::Error (optionally with source()). For each variant that wraps another error, you also implement From to enable ? propagation. This is entirely mechanical boilerplate — which is exactly why the crate exists.

A hand-written error enum with two variants covering a parse error and an I/O error, showing all the trait implementations needed to use it with ?.

use std::fmt;
use std::num::ParseIntError;

#[derive(Debug)]
enum AppError {
    Parse(ParseIntError),
    NegativeNumber(i32),
}

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::Parse(e) => write!(f, "parse error: {e}"),
            AppError::NegativeNumber(n) => write!(f, "expected positive, got {n}"),
        }
    }
}

impl std::error::Error for AppError {
    fn source(&self) -> Option<&(dyn std::error::Error + 'static)> {
        match self {
            AppError::Parse(e) => Some(e),
            _ => None,
        }
    }
}

// `From` makes `?` work when calling functions that return ParseIntError.
impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self {
        AppError::Parse(e)
    }
}

fn parse_positive(s: &str) -> Result<u32, AppError> {
    let n: i32 = s.trim().parse()?; // `?` converts ParseIntError -> AppError via From
    if n < 0 {
        return Err(AppError::NegativeNumber(n));
    }
    Ok(n as u32)
}

fn main() {
    println!("{:?}", parse_positive("42"));   // Ok(42)
    println!("{:?}", parse_positive("-5"));   // Err(NegativeNumber(-5))
    println!("{:?}", parse_positive("abc"));  // Err(Parse(...))
}

thiserror: Typed Errors for Libraries

thiserror is a derive macro that generates all that boilerplate for you. Instead of manually implementing Display, Error, and From, you annotate your enum variants with #[error("...")] attributes and derive thiserror::Error. The result is identical to the manual version but far more concise and less prone to drift (the display string lives right next to the variant it describes).

The #[error("...")] attribute accepts format strings where {field_name} expands to a field value and {0} expands to the first unnamed field. The #[from] attribute automatically generates the From impl, enabling ? propagation. The #[source] attribute marks a field as the underlying cause exposed by Error::source().

thiserror is the right tool when you are writing a library or any code whose error type is part of the public API. Your callers need to be able to match on the variants, inspect the fields, and make decisions based on the kind of error. A typed enum gives them the full picture; a boxed trait object or an opaque string does not.

Basic thiserror Usage

The same AppError rewritten with thiserror, eliminating all manual trait implementations while preserving identical behaviour.

// Cargo.toml:
// [dependencies]
// thiserror = "1"

use thiserror::Error;
use std::num::ParseIntError;

#[derive(Debug, Error)]
enum AppError {
    #[error("parse error: {0}")]
    Parse(#[from] ParseIntError),

    #[error("expected positive number, got {0}")]
    NegativeNumber(i32),

    #[error("value {value} is out of range [{min}, {max}]")]
    OutOfRange { value: i32, min: i32, max: i32 },
}

fn parse_in_range(s: &str, min: i32, max: i32) -> Result<i32, AppError> {
    let n: i32 = s.trim().parse()?; // ? uses From<ParseIntError> auto-generated by #[from]
    if n < min || n > max {
        return Err(AppError::OutOfRange { value: n, min, max });
    }
    Ok(n)
}

fn main() {
    println!("{:?}", parse_in_range("42", 0, 100));    // Ok(42)
    println!("{}",   parse_in_range("150", 0, 100).unwrap_err()); // value 150 is out of range [0, 100]
    println!("{}",   parse_in_range("abc", 0, 100).unwrap_err()); // parse error: ...
}

Error Chains and Sources

Real applications compose errors: a high-level operation like "load config file" fails because of a low-level I/O error. thiserror preserves this chain. The #[source] attribute marks which field holds the underlying cause; #[from] implies #[source] automatically. You can traverse the chain at runtime with Error::source(), or format the entire chain with a loop.

A layered error hierarchy: a low-level IoError, a mid-level ConfigError that wraps it, and a top-level StartupError that wraps ConfigError — all linked via #[from]/#[source].

use thiserror::Error;
use std::io;

#[derive(Debug, Error)]
enum ConfigError {
    #[error("cannot read config file '{path}': {source}")]
    Io {
        path: String,
        #[source]
        source: io::Error,
    },
    #[error("missing required key: {0}")]
    MissingKey(String),
    #[error("invalid value for '{key}': {message}")]
    InvalidValue { key: String, message: String },
}

#[derive(Debug, Error)]
enum StartupError {
    #[error("configuration failed")]
    Config(#[from] ConfigError),
    #[error("database connection failed: {0}")]
    Database(String),
}

fn load_config(path: &str) -> Result<String, ConfigError> {
    // Simulate a missing file
    Err(ConfigError::Io {
        path: path.to_string(),
        source: io::Error::new(io::ErrorKind::NotFound, "no such file"),
    })
}

fn start_app() -> Result<(), StartupError> {
    let _config = load_config("/etc/app/config.toml")?; // ? converts ConfigError -> StartupError
    Ok(())
}

fn main() {
    match start_app() {
        Ok(_) => println!("started"),
        Err(e) => {
            println!("error: {e}");
            // Traverse the error chain
            let mut source = e.source();
            while let Some(cause) = source {
                println!("  caused by: {cause}");
                source = cause.source();
            }
        }
    }
}

anyhow: Ergonomic Error Propagation in Applications

anyhow takes the opposite approach to thiserror. Rather than defining precise error types, it provides a single anyhow::Error type that can hold any error implementing std::error::Error, along with a Result<T> type alias (anyhow::Result<T> = Result<T, anyhow::Error>). The ? operator converts any compatible error into anyhow::Error automatically. No From implementations, no error enums, no boilerplate.

This makes anyhow the right tool for application code: CLI tools, server handlers, scripts, integration tests, and anywhere that calls into multiple libraries each with their own error types. You need to propagate those errors up to some top-level handler that will log them or display them to the user — you do not need to match on them. anyhow makes that propagation completely frictionless.

The golden rule: use thiserror in library code where callers need to inspect errors; use anyhow in application code where you just need to propagate and display them.

Basic anyhow Usage

Three functions that return different error types, all propagated through a single anyhow::Result return type with no adapter code.

// Cargo.toml:
// [dependencies]
// anyhow = "1"

use anyhow::{Result, Context, bail, ensure, anyhow};
use std::fs;

fn read_number_from_file(path: &str) -> Result<i32> {
    // fs::read_to_string returns io::Error; `?` converts it to anyhow::Error
    let contents = fs::read_to_string(path)
        .with_context(|| format!("failed to read '{path}'"))?;

    // str::parse returns ParseIntError; same conversion, no From impl needed
    let n: i32 = contents.trim().parse()
        .with_context(|| format!("'{path}' does not contain a valid integer"))?;

    Ok(n)
}

fn validate_port(port: i32) -> Result<u16> {
    // `ensure!` is like assert but returns an Err instead of panicking
    ensure!(port > 0 && port <= 65535, "port {port} is out of valid range [1, 65535]");
    Ok(port as u16)
}

fn create_error_example() -> Result<()> {
    // `bail!` is a shorthand for `return Err(anyhow!(...))`
    bail!("something went unexpectedly wrong");
}

fn main() -> Result<()> {
    match read_number_from_file("port.txt") {
        Ok(n)  => println!("port from file: {n}"),
        Err(e) => println!("could not read port: {e:#}"), // {:#} prints full chain
    }

    match validate_port(99999) {
        Ok(p)  => println!("port: {p}"),
        Err(e) => println!("invalid: {e}"),
    }

    Ok(())
}

Context and Error Messages

anyhow's killer feature is .context() and .with_context(). These methods wrap an error with an additional message, creating a chain that reads naturally from high-level to low-level cause. The difference between the two: .context("msg") always evaluates the string; .with_context(|| format!("...")) is lazy and only evaluates the closure when an error actually occurs. Use with_context when the message involves expensive formatting or allocations.

When displaying an anyhow::Error, use {:#} (alternate format) to print the full error chain separated by ": ", or use {:?} to get a debug representation that includes backtraces when RUST_BACKTRACE=1 is set.

A multi-step pipeline where each step adds contextual information with with_context, building up a rich error chain visible in the final message.

use anyhow::{Result, Context};

fn parse_config_value(raw: &str) -> Result<u32> {
    raw.trim()
        .parse::<u32>()
        .context("expected an unsigned integer")
}

fn load_config_line(line: &str) -> Result<(String, u32)> {
    let parts: Vec<&str> = line.splitn(2, '=').collect();
    if parts.len() != 2 {
        anyhow::bail!("expected 'key=value', got: {line:?}");
    }
    let key = parts[0].trim().to_string();
    let val = parse_config_value(parts[1])
        .with_context(|| format!("in key '{key}'"))?;
    Ok((key, val))
}

fn process_config(input: &str) -> Result<Vec<(String, u32)>> {
    input.lines()
         .filter(|l| !l.trim().is_empty())
         .map(|line| {
             load_config_line(line)
                 .with_context(|| format!("while parsing line: {line:?}"))
         })
         .collect()
}

fn main() {
    let good = "timeout = 30\nmax_retries = 5";
    let bad  = "timeout = 30\nmax_retries = abc";

    match process_config(good) {
        Ok(pairs) => println!("{pairs:?}"),
        Err(e) => println!("error: {e:#}"),
    }

    match process_config(bad) {
        Ok(pairs) => println!("{pairs:?}"),
        Err(e) => {
            println!("error: {e:#}");
            // prints: while parsing line: "max_retries = abc": in key 'max_retries': expected an unsigned integer: invalid digit found in string
        }
    }
}

Mixing thiserror and anyhow

In a real project you will often use both: thiserror in the library layers that define domain-specific error types, and anyhow in the application layer that glues everything together. The transition is seamless because any type implementing std::error::Error — including thiserror-derived types — converts automatically into anyhow::Error via ?.

A library module using thiserror for typed errors consumed by an application layer using anyhow for frictionless propagation.

// ---- library layer (uses thiserror) ----
mod db {
    use thiserror::Error;

    #[derive(Debug, Error)]
    pub enum DbError {
        #[error("connection refused: {0}")]
        ConnectionRefused(String),
        #[error("query failed: {0}")]
        QueryFailed(String),
        #[error("record not found: id={0}")]
        NotFound(u64),
    }

    pub fn find_user(id: u64) -> Result<String, DbError> {
        if id == 0 {
            return Err(DbError::NotFound(id));
        }
        Ok(format!("user_{id}"))
    }
}

// ---- application layer (uses anyhow) ----
use anyhow::{Result, Context};

fn get_user_name(id: u64) -> Result<String> {
    // DbError automatically converts to anyhow::Error via ?
    let name = db::find_user(id)
        .with_context(|| format!("failed to look up user {id}"))?;
    Ok(name)
}

fn main() -> Result<()> {
    println!("{}", get_user_name(42)?);

    match get_user_name(0) {
        Ok(name) => println!("{name}"),
        Err(e) => println!("error: {e:#}"),
    }

    // Application code can still downcast to the original error type when needed
    match db::find_user(0) {
        Err(db::DbError::NotFound(id)) => println!("no user with id {id}"),
        Err(e) => println!("other db error: {e}"),
        Ok(u)  => println!("{u}"),
    }

    Ok(())
}

Error Handling Patterns

Converting and Mapping Errors

Sometimes you need to convert one error type to another without From being defined, or you want to transform an error value before propagating it. Result::map_err applies a function to the error variant, leaving the Ok variant untouched. This is useful for wrapping third-party errors into your own error type without implementing From, or for adding runtime context to an existing error value.

Using map_err and ok_or_else to convert and enrich errors inline without implementing From.

use std::num::ParseIntError;

#[derive(Debug)]
struct AppError(String);

fn parse_positive_id(s: &str) -> Result<u32, AppError> {
    // map_err converts ParseIntError into AppError inline
    let n: i64 = s.trim()
        .parse()
        .map_err(|e: ParseIntError| AppError(format!("bad id '{s}': {e}")))?;

    // ok_or_else converts Option to Result
    u32::try_from(n)
        .ok()
        .filter(|&v| v > 0)
        .ok_or_else(|| AppError(format!("id must be positive, got {n}")))
}

fn main() {
    println!("{:?}", parse_positive_id("42"));   // Ok(42)
    println!("{:?}", parse_positive_id("0"));    // Err(AppError("id must be positive, got 0"))
    println!("{:?}", parse_positive_id("abc"));  // Err(AppError("bad id 'abc': ..."))
}

Collecting Results

A common pattern is processing a collection of inputs where each operation can fail, and you want either all the successes or the first failure. The Iterator::collect method supports collecting into Result<Vec<T>, E> directly — it short-circuits on the first Err and returns it, or returns Ok(Vec<T>) if all elements succeeded. If you want to collect all results instead of short-circuiting, use partition or a manual fold.

Two patterns: short-circuit collect into Result<Vec<T>>, and partition-based collection that gathers all errors without stopping early.

fn parse_numbers(inputs: &[&str]) -> Result<Vec<i32>, String> {
    inputs.iter()
        .map(|s| s.trim().parse::<i32>()
             .map_err(|e| format!("cannot parse '{s}': {e}")))
        .collect() // stops at first Err
}

fn parse_all_numbers(inputs: &[&str]) -> (Vec<i32>, Vec<String>) {
    let (oks, errs): (Vec<_>, Vec<_>) = inputs.iter()
        .map(|s| s.trim().parse::<i32>()
             .map_err(|e| format!("cannot parse '{s}': {e}")))
        .partition(Result::is_ok);

    let values: Vec<i32>   = oks.into_iter().map(Result::unwrap).collect();
    let errors: Vec<String> = errs.into_iter().map(Result::unwrap_err).collect();
    (values, errors)
}

fn main() {
    println!("{:?}", parse_numbers(&["1", "2", "3"]));      // Ok([1, 2, 3])
    println!("{:?}", parse_numbers(&["1", "x", "3"]));      // Err("cannot parse 'x': ...")

    let (vals, errs) = parse_all_numbers(&["1", "x", "3", "y"]);
    println!("ok: {vals:?}, errors: {errs:?}");
    // ok: [1, 3], errors: ["cannot parse 'x': ...", "cannot parse 'y': ..."]
}

When to Panic vs. Return an Error

Rust gives you two mechanisms for signalling failure: panic! (for bugs and violated invariants) and Result/Option (for expected, recoverable failures). The decision of which to use is a design choice about the nature of the failure:

  • Return Result when the failure is expected and the caller can reasonably recover: file not found, network timeout, malformed user input, API rate limit. These are not bugs; they are part of normal operation.
  • Use panic! when the failure represents a programming error that should never occur in correct code: a violated precondition, an index out of bounds that should be unreachable given prior validation, or an impossible enum variant. Panics signal bugs, not runtime conditions.
  • Use unwrap() / expect() sparingly, in tests and prototypes only. In production code, prefer ? or explicit match. The one exception: expect("invariant X should hold because Y") is acceptable when you have a logical proof that the Option/Result cannot be None/Err at that point, and you want a clear message if your reasoning was wrong.

A function pair illustrating correct placement of panic vs Result: panicking on violated invariants, returning Result for expected runtime failures.

use anyhow::{Result, Context, ensure};

/// Panics if `data` is empty -- this is a programmer error; callers must validate.
fn median(data: &[f64]) -> f64 {
    assert!(!data.is_empty(), "median() called with empty slice -- this is a bug");
    let mut sorted = data.to_vec();
    sorted.sort_by(f64::total_cmp);
    let mid = sorted.len() / 2;
    if sorted.len() % 2 == 0 {
        (sorted[mid - 1] + sorted[mid]) / 2.0
    } else {
        sorted[mid]
    }
}

/// Returns Err for expected failures: bad format, out of range input.
fn parse_sample_list(s: &str) -> Result<Vec<f64>> {
    ensure!(!s.trim().is_empty(), "input is empty");
    s.split(',')
        .map(|tok| tok.trim().parse::<f64>()
             .with_context(|| format!("invalid token: {tok:?}")))
        .collect()
}

fn main() {
    match parse_sample_list("1.5, 2.5, 3.0, 4.0") {
        Ok(data) => println!("median: {}", median(&data)),
        Err(e)   => println!("error: {e:#}"),
    }

    match parse_sample_list("1.5, two, 3.0") {
        Ok(data) => println!("median: {}", median(&data)),
        Err(e)   => println!("error: {e:#}"),
        // error: invalid token: "two": invalid float literal
    }
}
