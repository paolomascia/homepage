Smart Pointers: Box, Rc & Arc

Rust Smart Pointers Memory

Rust's ownership system handles the majority of memory management automatically. Smart pointers extend this system to cover cases the basic rules cannot express: heap allocation when the size is unknown at compile time (Box), shared ownership without a garbage collector (Rc), and shared ownership across threads (Arc). Each smart pointer implements Deref so it can be used transparently wherever a reference to the inner type is expected.

Box<T> — Heap Allocation

Box<T> allocates a value of type T on the heap and gives you a uniquely owned pointer to it. When the Box goes out of scope, it drops the value and frees the heap memory. There is no reference counting, no garbage collection: the drop is deterministic and happens exactly when the binding is released.

The three canonical reasons to use Box:

  1. Recursive types: a type whose size cannot be computed because it contains itself directly. Wrapping the recursive field in Box gives it a known size (one pointer).
  2. Trait objects: Box<dyn Trait> stores a dynamically dispatched value whose concrete type is erased (covered in the trait objects article).
  3. Large values: moving a large struct copies it on the stack. Boxing it puts it on the heap and makes the move free (just a pointer copy).

Box Basics and Recursive Types

A recursive linked list and binary tree that require Box to have a known size, plus the Deref coercion that makes Box transparent.

// Without Box: compile error -- `List` has infinite size.
// enum List { Cons(i32, List), Nil }

// With Box: size is known (one pointer + one i32).
#[derive(Debug)]
enum List {
    Cons(i32, Box<List>),
    Nil,
}

impl List {
    fn new() -> Self { List::Nil }

    fn prepend(self, value: i32) -> Self {
        List::Cons(value, Box::new(self))
    }

    fn sum(&self) -> i32 {
        match self {
            List::Nil => 0,
            List::Cons(v, rest) => v + rest.sum(),
        }
    }
}

// Binary tree -- each node has two optional boxed children.
#[derive(Debug)]
struct TreeNode {
    value: i32,
    left:  Option<Box<TreeNode>>,
    right: Option<Box<TreeNode>>,
}

impl TreeNode {
    fn leaf(value: i32) -> Self { TreeNode { value, left: None, right: None } }

    fn insert(mut self, v: i32) -> Self {
        if v < self.value {
            self.left = Some(Box::new(match self.left {
                None      => TreeNode::leaf(v),
                Some(l)   => l.insert(v),
            }));
        } else {
            self.right = Some(Box::new(match self.right {
                None      => TreeNode::leaf(v),
                Some(r)   => r.insert(v),
            }));
        }
        self
    }

    fn inorder(&self) -> Vec<i32> {
        let mut result = Vec::new();
        if let Some(l) = &self.left  { result.extend(l.inorder()); }
        result.push(self.value);
        if let Some(r) = &self.right { result.extend(r.inorder()); }
        result
    }
}

fn main() {
    // Linked list
    let list = List::new().prepend(3).prepend(2).prepend(1);
    println!("{:?}", list);  // Cons(1, Cons(2, Cons(3, Nil)))
    println!("sum: {}", list.sum()); // 6

    // Box: Deref coercion makes it transparent
    let boxed: Box<i32> = Box::new(42);
    println!("{}", *boxed);      // 42 -- explicit deref
    println!("{}", boxed + 8);   // 50 -- auto-deref in arithmetic

    // BST
    let mut tree = TreeNode::leaf(5);
    for v in [3, 7, 1, 4, 6, 8] { tree = tree.insert(v); }
    println!("inorder: {:?}", tree.inorder()); // [1, 3, 4, 5, 6, 7, 8]
}

Box for Large Values and Trait Objects

Moving a large struct without copying, and boxing trait objects for heterogeneous storage.

// A large struct -- moving it without Box copies 4KB on the stack.
struct LargeMatrix([[f64; 64]; 64]);

impl LargeMatrix {
    fn new() -> Self { LargeMatrix([[0.0; 64]; 64]) }
    fn set(&mut self, r: usize, c: usize, v: f64) { self.0[r][c] = v; }
}

trait Drawable { fn draw(&self); }

struct Circle   { r: f64 }
struct Square   { s: f64 }
impl Drawable for Circle { fn draw(&self) { println!("circle r={}", self.r); } }
impl Drawable for Square { fn draw(&self) { println!("square s={}", self.s); } }

fn main() {
    // Box keeps LargeMatrix on the heap; moving the Box is a single pointer copy.
    let matrix = Box::new(LargeMatrix::new());
    let _moved = matrix; // cheap: moves 8-byte pointer, not 32KB

    // Heterogeneous collection via Box<dyn Trait>
    let shapes: Vec<Box<dyn Drawable>> = vec![
        Box::new(Circle { r: 3.0 }),
        Box::new(Square { s: 4.0 }),
        Box::new(Circle { r: 1.5 }),
    ];
    for shape in &shapes { shape.draw(); }

    // Box::leak: intentionally leak memory to get a &'static reference.
    // Use only for values that truly need to live forever (e.g., global config).
    let config: &'static str = Box::leak(Box::new("static config".to_string()));
    println!("{config}");
}

Rc<T> — Shared Ownership (Single-Threaded)

Rc<T> (Reference Counted) enables multiple owners of the same heap value within a single thread. Internally it stores the value alongside two counters: the strong reference count and the weak reference count. Cloning an Rc increments the strong count; dropping one decrements it. When the strong count reaches zero, the value is dropped.

The critical constraint: Rc is not Send (cannot be moved across threads) and not Sync (cannot be shared across threads). Its reference counting is non-atomic, which is fast but unsafe to do from multiple threads. For multi-threaded shared ownership, use Arc.

Typical use cases: graph structures where nodes share edges, caches where multiple parts of the code hold references to the same value, and tree nodes that need to point to their parent without creating ownership cycles (use Weak for the back-pointer).

Rc Basics and Reference Counts

Creating shared owners, inspecting reference counts, and the automatic drop when the last owner is gone.

use std::rc::Rc;

#[derive(Debug)]
struct Node {
    value: i32,
    children: Vec<Rc<Node>>,
}

impl Node {
    fn new(value: i32) -> Rc<Self> {
        Rc::new(Node { value, children: Vec::new() })
    }
}

fn main() {
    let a = Rc::new(42);
    println!("strong count after creating a: {}", Rc::strong_count(&a)); // 1

    let b = Rc::clone(&a);  // increments count, does NOT copy the value
    println!("strong count after cloning:    {}", Rc::strong_count(&a)); // 2

    {
        let c = Rc::clone(&a);
        println!("strong count with c:         {}", Rc::strong_count(&a)); // 3
        println!("a={}, b={}, c={}", a, b, c); // all print 42
    } // c dropped here

    println!("strong count after c dropped:  {}", Rc::strong_count(&a)); // 2

    // Shared ownership: both owner1 and owner2 point to the same Config
    #[derive(Debug)]
    struct Config { debug: bool, max_connections: u32 }

    let config = Rc::new(Config { debug: true, max_connections: 100 });
    let server_config = Rc::clone(&config);
    let monitor_config = Rc::clone(&config);

    println!("server uses debug={}", server_config.debug);
    println!("monitor uses max_conn={}", monitor_config.max_connections);
    println!("shared config refs: {}", Rc::strong_count(&config)); // 3

    // Rc::try_unwrap: get the inner value if this is the only owner
    let sole = Rc::new("only owner".to_string());
    match Rc::try_unwrap(sole) {
        Ok(s)  => println!("unwrapped: {s}"),
        Err(_) => println!("still shared"),
    }
}

Weak<T> — Breaking Reference Cycles

Reference cycles between Rc values cause memory leaks: if A holds an Rc to B and B holds an Rc to A, neither strong count ever reaches zero. Weak<T> solves this: a weak reference does not increment the strong count and does not keep the value alive. To access the value through a Weak, call .upgrade() which returns Option<Rc<T>> — None if the value has already been dropped.

The idiomatic pattern for parent-child trees: children hold strong Rc references to their parents, and parents hold Weak references back to their children (or vice versa, depending on ownership semantics).

A parent-child tree where children use Weak back-pointers to avoid cycles, with upgrade() to access the parent.

use std::rc::{Rc, Weak};
use std::cell::RefCell;

#[derive(Debug)]
struct TreeNode {
    value:    i32,
    parent:   RefCell<Weak<TreeNode>>,   // weak: does not own parent
    children: RefCell<Vec<Rc<TreeNode>>>, // strong: owns children
}

impl TreeNode {
    fn new(value: i32) -> Rc<Self> {
        Rc::new(TreeNode {
            value,
            parent:   RefCell::new(Weak::new()),
            children: RefCell::new(Vec::new()),
        })
    }
}

fn main() {
    let root = TreeNode::new(1);
    let child1 = TreeNode::new(2);
    let child2 = TreeNode::new(3);

    // Set child's parent (weak reference -- does not keep root alive)
    *child1.parent.borrow_mut() = Rc::downgrade(&root);
    *child2.parent.borrow_mut() = Rc::downgrade(&root);

    // Add children to root
    root.children.borrow_mut().push(Rc::clone(&child1));
    root.children.borrow_mut().push(Rc::clone(&child2));

    // Strong count: root has 1 (the binding) + 2 (children hold it via Weak, not strong)
    println!("root strong count: {}", Rc::strong_count(&root)); // 1

    // Upgrade weak reference: child1 can access its parent
    if let Some(parent) = child1.parent.borrow().upgrade() {
        println!("child1's parent value: {}", parent.value); // 1
    }

    // After root is dropped, child.parent.upgrade() returns None
    let child1_ref = Rc::clone(&child1);
    drop(root);
    // root's strong count was 1, now 0 -- dropped
    println!("parent gone: {:?}", child1_ref.parent.borrow().upgrade()); // None
}

Arc<T> — Shared Ownership Across Threads

Arc<T> (Atomically Reference Counted) is Rc<T> with atomic operations on the reference counters, making it safe to clone and drop from multiple threads simultaneously. The atomic operations use hardware compare-and-swap instructions, which are more expensive than non-atomic increments but still very fast in the uncontended case.

Arc<T> is Send + Sync when T: Send + Sync, meaning it can be moved across thread boundaries and shared via references. The value inside is immutable by default; combine with Mutex or RwLock for thread-safe mutation (covered in the next article).

Arc in Multi-Threaded Code

Sharing read-only data across threads with Arc, and the classic Arc<Mutex<T>> pattern for shared mutable state.

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Read-only shared data: Arc without Mutex
    let data = Arc::new(vec![1, 2, 3, 4, 5]);

    let handles: Vec<_> = (0..4).map(|i| {
        let data = Arc::clone(&data);
        thread::spawn(move || {
            // Each thread reads from the shared Vec
            let sum: i32 = data.iter().sum();
            println!("thread {i}: sum = {sum}");
        })
    }).collect();

    for h in handles { h.join().unwrap(); }

    // Shared mutable state: Arc<Mutex<T>>
    let counter = Arc::new(Mutex::new(0u32));

    let handles: Vec<_> = (0..8).map(|_| {
        let counter = Arc::clone(&counter);
        thread::spawn(move || {
            let mut val = counter.lock().unwrap();
            *val += 1;
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
    println!("final counter: {}", counter.lock().unwrap()); // 8

    // Arc with a complex shared structure
    #[derive(Debug)]
    struct SharedCache {
        data: Mutex<std::collections::HashMap<String, String>>,
    }

    impl SharedCache {
        fn new() -> Arc<Self> {
            Arc::new(SharedCache { data: Mutex::new(std::collections::HashMap::new()) })
        }
        fn set(&self, key: &str, val: &str) {
            self.data.lock().unwrap().insert(key.to_string(), val.to_string());
        }
        fn get(&self, key: &str) -> Option<String> {
            self.data.lock().unwrap().get(key).cloned()
        }
    }

    let cache = SharedCache::new();
    let cache2 = Arc::clone(&cache);

    let writer = thread::spawn(move || {
        cache2.set("key", "value");
    });
    writer.join().unwrap();

    println!("{:?}", cache.get("key")); // Some("value")
}

Deref and DerefMut — Transparent Pointer Behaviour

All smart pointers implement Deref (and DerefMut where appropriate), which is what allows them to be used transparently wherever a reference to the inner type is expected. The compiler applies deref coercion automatically: when it sees &Box<T> where &T is required, it inserts the implicit * operations needed to coerce the type. This chain can go through multiple levels (e.g., &String → &str, &Box<String> → &String → &str).

Custom smart pointer implementing Deref, and the deref coercion chain in action.

use std::ops::{Deref, DerefMut};

// A minimal smart pointer wrapping a value
struct MyBox<T>(T);

impl<T> MyBox<T> {
    fn new(val: T) -> Self { MyBox(val) }
}

impl<T> Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T { &self.0 }
}

impl<T> DerefMut for MyBox<T> {
    fn deref_mut(&mut self) -> &mut T { &mut self.0 }
}

fn print_str(s: &str) { println!("{s}"); }

fn main() {
    let b = MyBox::new(String::from("hello"));

    // Explicit deref: *b gives String, &*b gives &String, which coerces to &str
    print_str(&*b);  // explicit

    // Implicit deref coercion: &MyBox<String> -> &String -> &str
    print_str(&b);   // compiler inserts the derefs automatically

    // Standard library: Box, Rc, Arc all work the same way
    let boxed = Box::new(String::from("boxed"));
    print_str(&boxed); // &Box<String> -> &String -> &str

    // Mutation via DerefMut
    let mut mb = MyBox::new(5i32);
    *mb += 10;
    println!("{}", *mb); // 15

    // Deref coercion summary:
    // &T          -> &T           (identity)
    // &Box<T>     -> &T           (Box deref)
    // &String     -> &str         (String deref)
    // &Vec<T>     -> &[T]         (Vec deref)
    // &Box<String> -> &String -> &str  (two steps, both auto)
}

Choosing the Right Pointer Type

A quick decision guide:

  • Stack allocation (no smart pointer): default. If the value fits on the stack and has a known size, just use it directly.
  • Box<T>: single owner, heap-allocated. Use for recursive types, large values, or trait objects. Zero overhead beyond the heap allocation itself.
  • Rc<T>: multiple owners, single thread. Use when ownership is shared and lifetimes cannot be expressed statically. Slightly more expensive than Box due to reference count maintenance.
  • Weak<T>: non-owning reference into an Rc. Use for back-pointers and cyclic structures to prevent memory leaks.
  • Arc<T>: multiple owners, multiple threads. Use when shared data must cross thread boundaries. Slightly more expensive than Rc due to atomic operations.
  • Arc<Mutex<T>>: shared mutable state across threads. The most common pattern for concurrent data sharing.
