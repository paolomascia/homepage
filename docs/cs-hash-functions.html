Hash Functions

Cryptography Hashing MAC

Cryptographic hash functions serve as fundamental tools in security protocols. They generate a fixed-length “digest” from input of any size—essentially a message fingerprint. Unlike encryption systems, hashes are keyless. They are vital for digital signatures, message verification, secure password storage (using specialized slow hashing), and key generation.

Security Properties

Three core requirements define secure hash functions:

  • Preimage resistance (one-wayness): Given output z, finding input x where h(x) = z should be computationally impossible.
  • Second-preimage resistance: Finding a different x′ producing identical output to a known x should be infeasible.
  • Collision resistance: Discovering any two distinct inputs yielding the same digest should be impractical.

Common applications include fingerprinting, digital signature generation, message authentication codes (HMAC), and key derivation functions (KDF).

MD5

MD5 produces 128-bit digests but is not cryptographically secure and should serve only as a non-cryptographic checksum for detecting accidental corruption.

Its workflow involves five stages:

  1. Padding the message
  2. Appending length information
  3. Initializing four 32-bit state variables
  4. Processing 512-bit blocks through four rounds using specific Boolean functions
  5. Outputting the concatenated final state
// MD5 high-level structure (conceptual)
// Initialize state
A = 0x67452301;
B = 0xEFCDAB89;
C = 0x98BADCFE;
D = 0x10325476;

// Process each 512-bit block
for each block M:
    // 4 rounds of 16 operations each
    // Round 1: F(B,C,D) = (B & C) | (~B & D)
    // Round 2: G(B,C,D) = (B & D) | (C & ~D)
    // Round 3: H(B,C,D) = B ^ C ^ D
    // Round 4: I(B,C,D) = C ^ (B | ~D)

// Output: A || B || C || D  (128 bits)

MD5’s weaknesses include rapid computation enabling brute-force attacks, widespread precomputed rainbow tables, and broken collision resistance allowing practical chosen-prefix collision construction. For passwords, dedicated functions like bcrypt, scrypt, or Argon2 are mandatory—never use MD5 even with salt.

SHA Family

NIST standardizes the SHA family (FIPS 180-4), including deprecated SHA-1 (160-bit) and modern SHA-2 variants (SHA-224, SHA-256, SHA-384, SHA-512). SHA-3 (Keccak), selected through public competition in 2012, offers an alternative sponge-based architecture.

SHA-1 is deprecated due to practical collision attacks. SHA-2 remains widely trusted, using 32-bit or 64-bit word operations depending on variant. SHA-3 uses sponge construction enabling variable-length output through SHAKE128/256 functions.

// SHA-256 high-level structure (conceptual)
// Initialize 8 x 32-bit state words (H0..H7)
// from first 32 bits of fractional parts of
// square roots of first 8 primes

// Process each 512-bit block:
//   Expand 16 message words to 64 schedule words
//   64 rounds of compression using:
//     - Ch(e,f,g), Maj(a,b,c) functions
//     - Sigma rotations
//     - Round constants K[0..63]

// Output: H0 || H1 || ... || H7  (256 bits)

MAC (Message Authentication Code)

MACs provide integrity and authenticity using shared secret keys. The sender generates a tag from the key and message; receivers verify by recomputing and comparing tags.

Construction methods include:

  • HMAC: Hash-based, built from cryptographic hashes like HMAC-SHA-256.
  • CBC-MAC / CMAC: Block-cipher-based approaches.
  • PMAC / GMAC: Parallelizable or Galois-field variants.
// HMAC construction (conceptual)
// HMAC(K, m) = H( (K' ^ opad) || H( (K' ^ ipad) || m ) )
//
// K'    = key padded/hashed to block size
// ipad  = 0x36 repeated to block size
// opad  = 0x5C repeated to block size
// H     = underlying hash function (e.g., SHA-256)
//
// Two-pass hash provides resistance to
// length-extension attacks

Best practice combines encryption with authentication through AEAD modes (AES-GCM, ChaCha20-Poly1305). When AEAD is unavailable, encrypt-then-MAC is recommended—computing the MAC over ciphertext rather than plaintext first.

Practical Recommendations

  • Prefer SHA-256/384/512 or SHA-3; avoid MD5 and SHA-1 for security tasks.
  • Use bcrypt, scrypt, or Argon2 for passwords with per-password salt and cost parameters.
  • Employ HMAC when AEAD is unavailable; otherwise use AEAD schemes.
