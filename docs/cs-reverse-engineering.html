Reverse Engineering

Reverse Engineering Malware Analysis Cybersecurity

Reverse engineering studies compiled software to determine design, control flow, and behaviors — including concealed or malicious ones. It “reverses” the development process: from executable artifacts back to useful representations like assembly, intermediate representation, or decompiled source-like code. This comprehensive guide covers core tooling, executable formats, malware analysis workflows, and the ethical principles that govern the practice.

Disclaimer: This resource is intended for educational and defensive cybersecurity purposes. Techniques described should only be used in authorized testing, research, or forensic contexts. Unauthorized software analysis may breach laws or licensing agreements — always obtain written permission.

The Challenge of Reverse Engineering

Reverse engineering studies compiled software to determine design, control flow, and behaviors — including concealed or malicious ones. It “reverses” the development process: from executable artifacts back to useful representations like assembly, intermediate representation, or decompiled source-like code.

Core Tooling Paradigms

  • Decompilers: Attempt to rebuild high-level constructs (functions, loops, types) from binaries into C/C++/C#-like code. Output is approximate and may not perfectly match the original source.
  • Disassemblers: Convert machine code to assembly (ISA-specific), maintaining precise control flow and operands but without high-level structure unless inferred by the tool.

Why Reverse Engineering Is Hard

  • Compilation is lossy: Symbol names, comments, types, and abstractions are often stripped or optimized away during compilation.
  • Many-to-many mapping: Multiple source patterns compile to similar machine code; conversely, one source construct can compile differently depending on flags and optimizations.
  • Compiler/ABI variation: Calling conventions, inlining, link-time optimization (LTO), and optimization levels change the shape of emitted code.
  • Anti-RE measures: Packing, obfuscation, VM-based protectors, control flow flattening, opaque predicates, and anti-debug techniques all complicate analysis.

Programming Languages & Executable Production

By Abstraction Level

  • First Generation (machine code): Raw opcodes and hex values; unreadable for humans.
  • Second Generation (assembly): Mnemonics that map 1:1 to opcodes; readable but low-level.
  • Third Generation (C, C++, Fortran, etc.): Portable and expressive; compiled to native code or hybrid formats.

By Execution Model

  • Interpreted: Source code executed directly by an interpreter; decompilation is moot (you have the source), but code can still be obfuscated.
  • Compiled: Compiler emits native machine code as object files and executables.
  • Hybrid: Compiled to bytecode/IL for a virtual machine (Java class → JVM, .NET IL → CLR). Rich metadata often eases reverse engineering.

Executable File Formats

Windows Portable Executable (PE/COFF)

PE packages code and data plus metadata for the Windows loader. “Portable” refers to supporting multiple architectures (x86, x64, ARM).

Layout (simplified):

  • MS-DOS header (MZ): Legacy compatibility header with a pointer to the PE signature.
  • PE header (PE\0\0): Machine type, number of sections, entry point, and timestamps.
  • Optional header: Sizes, subsystem type, ImageBase, AddressOfEntryPoint, and data directories (IAT, relocations, resources, TLS, etc.).
  • Section table + sections: Common names include .text (code), .rdata (imports/read-only data), .data (read-write data), .rsrc (resources), .reloc (relocations).

Import Address Table (IAT): A thunk table that the loader fills with function addresses from DLLs; inspecting the IAT reveals which APIs the binary uses.

Relocations: If the preferred base (ImageBase) is unavailable (common with ASLR), the loader applies relocations from .reloc to fix absolute addresses.

Resources: Dialogs, icons, manifests, and version information stored in .rsrc.

Debug symbols (optional): Separate .pdb files on Windows; when present, they greatly assist reverse engineering.

ELF (Executable and Linkable Format)

ELF is the standard binary format on Unix-like systems. It separates concerns for linking and runtime loading.

  • ELF Header: Architecture, endianness, entry point, and header offsets.
  • Program Headers (loader-facing): Segments to map into memory (e.g., PT_LOAD), dynamic section (PT_DYNAMIC), interpreter (PT_INTERP).
  • Section Headers (linker-facing): .text, .rodata, .data, .bss, .dynsym/.symtab, .rela.*, .plt/.got, .comment, etc.
  • PLT/GOT: Indirection tables for dynamic linking; useful for tracing external library calls.

Essential Reverse Engineering Tools

CLI / Quick-Triage

strings: Extract printable strings (useful for URLs, paths, error messages).

strings -a -n 4 sample.bin | less
strings -el sample.exe     # UTF-16LE (common on Windows)

objdump (ELF): Disassembly, headers, and relocations.

objdump -f -h -p sample.elf
objdump -d -M intel sample.elf | less
readelf -a sample.elf

dumpbin (PE, Windows): Headers, imports/exports, and disassembly.

dumpbin /headers /imports sample.exe
dumpbin /exports sample.dll
dumpbin /disasm sample.exe

PE tools: PE header parsers and viewers to inspect sections, IAT, relocations, and resources.

Interactive Suites

  • Ghidra: Free disassembler + decompiler from the NSA with powerful program analysis and scripting capabilities. Cross-platform (Java).
  • IDA Free/Pro: Industry-standard disassembler with decompiler modules (commercial license).
  • radare2 / Cutter: Open-source framework + GUI with analysis and decompilation features.

Installing Ghidra (Conceptual)

  1. Download the distribution (ZIP) from the official site.
  2. Unpack into a directory you control.
  3. Launch ghidraRun (Linux/macOS) or ghidraRun.bat (Windows).
  4. Point to a supported Java runtime if prompted; accept the license; create a project and import your sample.

Ghidra Interface Primer

  • CodeBrowser: Central view for listing, decompiler, data types, and references.
  • Listing: Disassembly (assembly view) with cross-references and function boundaries.
  • Decompiler: High-level pseudocode; rename variables, retype, and apply structures.
  • Symbol Tree: Functions, globals, and labels for quick navigation and organization.

Malware Analysis: Phases & Environments

Typical Workflow

  1. Identification/Triage: File type, packing, architecture, signatures (AV, YARA), strings, entropy analysis.
  2. Static Analysis: Headers, imports, control flow, and decompiled views — without executing the sample.
  3. Dynamic Analysis: Execute in an isolated sandbox/VM; monitor filesystem, registry, process tree, and network activity.

Sandboxes & Lab Stacks

  • Sandbox: Isolated VM(s) with instrumentation (snapshots, network sinkhole, TLS intercept for telemetry).
  • Cuckoo: Open-source dynamic analysis platform supporting Windows and Linux guests.
  • REMnux: Linux distribution curated for malware forensics and reverse engineering tooling.
  • FLARE VM: Windows VM toolkit preloaded for reverse engineering and dynamic analysis.

Scripts & Macros in the Threat Landscape

PowerShell

PowerShell scripts can administer systems — and be abused for “living-off-the-land” attacks. In reverse engineering, focus on decoding payloads, resolving Invoke-* download/execute patterns, and recovering IOCs (Indicators of Compromise).

  • ISE: Integrated Scripting Environment for composing and testing scripts.
  • Obfuscation techniques: Concatenation, Base64 encoding, compression, string character codes, whitespace removal, and staged loaders.

VBA Macros (Office)

Macros automate Office tasks, but attackers may embed staged payloads triggered by document events (e.g., AutoOpen). Analysis tools include:

  • oletools suite: oleid (risk flags), olevba (extract/deobfuscate macros), macrodump, emldump.

Managed Runtimes

.NET (CLR)

.NET assemblies (PE format) contain IL (Intermediate Language) and rich metadata (types, methods, properties). Reverse engineering is often high-fidelity due to the metadata richness.

  • Explore with dotPeek, ILSpy, or dnSpy (decompile to C#).
  • Inspect manifests, resources, and embedded assemblies; note strong-name and signing information.

Java (JAR/Class)

JARs are ZIP archives containing class files with JVM bytecode, constant pools, and attributes (LineNumber, LocalVariableTable).

  • Use JD-GUI, Bytecode Viewer, procyon, or cfr to decompile.

Safe, Systematic Tactics

  • Begin read-only: Hash the sample (e.g., SHA-256), record provenance, and copy to an isolated lab environment.
  • Check packing: Entropy spikes, few imports, and section names like .upx suggest packing — attempt benign unpacking first (if authorized).
  • Map imports: Network APIs (WinInet/WinHTTP), process management (CreateProcess), registry access, and crypto usage provide hints on the binary’s capabilities.
  • Follow the money: Strings and resources often reveal C2 domains, file names, mutexes, and campaign identifiers.
  • Respect legal boundaries: Licenses, EULAs, and local laws (e.g., anti-circumvention statutes) may apply even during research.

Worked Mini-Examples (Non-destructive)

Quick PE/ELF Headers

# ELF
readelf -h -lW sample.elf
objdump -x sample.elf

# PE (Windows)
dumpbin /headers sample.exe

Strings & Encodings

strings -a -n 6 sample.bin | grep -Ei 'http|dns|key|pass|user'
strings -el sample.exe | head

Function Map (ELF)

nm -C --defined-only sample.elf
objdump -d -M intel sample.elf | less

Common Obstacles & How to Think About Them

  • Packing/Compression: The binary unpacks at runtime; consider memory dumps after unpacking stubs execute (legally and safely in an isolated lab).
  • Obfuscation: Renaming, junk code, and control-flow flattening complicate analysis. Lean on the decompiler, control flow graph (CFG), and data-flow analysis; rename variables meaningfully as you work.
  • Anti-debug: Timing checks and API probes detect debugger presence. Countermeasures include VMs with stealth configurations, patching (in a lab only), and trace-based tools.
  • ASLR/PIE: Addresses differ per run; rely on relative offsets and symbols when available.
  • Stripped binaries: No symbols are present; infer function prototypes via calling convention patterns and import usage analysis.

Ethics and Compliance

  • Obtain explicit authorization; keep detailed notes and preserve chain-of-custody where forensics matter.
  • Prefer isolated networks (no bridge to production); sinkhole external communications.
  • Share indicators responsibly; follow coordinated disclosure when applicable.
  • The techniques discussed are intended for legitimate cybersecurity, research, and educational use. Always ensure proper authorization and comply with all applicable laws, licenses, and organizational policies before analyzing software.
