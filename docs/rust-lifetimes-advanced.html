Advanced Lifetimes & Higher-Rank Trait Bounds

Rust Lifetimes HRTB

Lifetime annotations are Rust's way of expressing how long references are valid. The basic rules — annotating a struct field or function return value to tie it to an input reference — are covered in the ownership article. This article goes deeper: named lifetime elision, lifetime subtyping, the 'static lifetime, variance, and Higher-Rank Trait Bounds (HRTB), which express constraints that must hold for all possible lifetimes rather than a specific one.

Lifetime Elision Rules

Rust applies three elision rules that let you omit explicit lifetime annotations in the most common cases. Understanding these rules lets you predict when annotations are required and when they can be omitted:

  1. Each reference parameter gets its own lifetime: fn f(x: &T, y: &U) → fn f<'a, 'b>(x: &'a T, y: &'b U).
  2. If there is exactly one input lifetime, it is assigned to all output lifetimes: fn f(x: &T) -> &U → fn f<'a>(x: &'a T) -> &'a U.
  3. If one of the inputs is &self or &mut self, its lifetime is assigned to all output lifetimes.

When these rules produce an unambiguous result, no annotations are needed. When they do not — typically when a function takes multiple references and returns one — you must be explicit.

Examples where elision applies (no annotations needed) and where it does not (annotations required), with the expanded form shown as comments.

// Elision rule 2: one input lifetime -> all outputs inherit it.
// Expanded: fn first_word<'a>(s: &'a str) -> &'a str
fn first_word(s: &str) -> &str {
    s.split_whitespace().next().unwrap_or("")
}

// Elision rule 3: &self -> output inherits self's lifetime.
struct Buffer(String);
impl Buffer {
    // Expanded: fn contents<'a>(&'a self) -> &'a str
    fn contents(&self) -> &str {
        &self.0
    }
}

// Elision FAILS: two input lifetimes, compiler can't choose which to assign.
// Must be explicit:
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

// Different input lifetimes: output only borrows from one.
fn first_of_two<'a, 'b>(x: &'a str, _y: &'b str) -> &'a str {
    x
}

fn main() {
    let s = String::from("hello world");
    println!("{}", first_word(&s)); // hello

    let b = Buffer("content".to_string());
    println!("{}", b.contents()); // content

    let s1 = String::from("longer string");
    let result;
    {
        let s2 = String::from("short");
        result = longest(&s1, &s2);
        println!("longest: {result}");
    }
    // `result` cannot be used here: it may point to s2 which is now dropped.
}

Lifetime Annotations in Structs

When a struct holds a reference, the struct's type must be parameterized by the lifetime of that reference. This ensures that no struct instance can outlive the data it borrows. The lifetime annotation is part of the struct's type, and the compiler enforces it at every point of use.

A Parser struct that borrows its input string, and a Token struct that borrows slices of it, showing how lifetime parameters flow through nested structs.

#[derive(Debug)]
struct Token<'a> {
    kind: TokenKind,
    text: &'a str,
}

#[derive(Debug)]
enum TokenKind { Word, Number, Punct }

struct Parser<'a> {
    input: &'a str,
    pos:   usize,
}

impl<'a> Parser<'a> {
    fn new(input: &'a str) -> Self {
        Parser { input, pos: 0 }
    }

    fn next_word(&mut self) -> Option<Token<'a>> {
        // Skip whitespace
        while self.pos < self.input.len()
            && self.input.as_bytes()[self.pos] == b' '
        {
            self.pos += 1;
        }
        if self.pos == self.input.len() { return None; }

        let start = self.pos;
        while self.pos < self.input.len()
            && self.input.as_bytes()[self.pos] != b' '
        {
            self.pos += 1;
        }
        let text = &self.input[start..self.pos];
        let kind = if text.chars().all(|c| c.is_numeric()) {
            TokenKind::Number
        } else {
            TokenKind::Word
        };
        Some(Token { kind, text })
    }
}

fn main() {
    let source = String::from("hello 42 world 99");
    let mut parser = Parser::new(&source);

    while let Some(token) = parser.next_word() {
        println!("{:?}", token);
    }
    // Token { kind: Word,   text: "hello" }
    // Token { kind: Number, text: "42"    }
    // Token { kind: Word,   text: "world" }
    // Token { kind: Number, text: "99"    }
}

Lifetime Subtyping

Lifetime subtyping is expressed as 'long: 'short, read "lifetime 'long outlives 'short." It tells the compiler that wherever a reference with 'short is required, a reference with 'long can be substituted, because 'long is guaranteed to remain valid for at least as long as 'short. This is analogous to subtyping in object-oriented languages: a longer-lived reference is a "subtype" of a shorter-lived reference because it is more capable (valid for longer).

You rarely need to write subtyping constraints explicitly; the compiler infers them. But in complex generic code with multiple lifetime parameters, explicit subtyping constraints allow the compiler to understand relationships it could not figure out from the function signature alone.

A function that requires a reference that lives at least as long as a given lifetime, demonstrating explicit subtyping bounds.

// 'long: 'short means 'long outlives 'short.
// The function can return &'short T from a &'long T because 'long >= 'short.
fn shorter_of<'short, 'long: 'short>(
    x: &'long str,
    y: &'short str,
) -> &'short str {
    if x.len() < y.len() { x } else { y }
}

// A cache that holds references whose lifetime 'data must outlive 'cache
struct Cache<'cache, 'data: 'cache> {
    entries: Vec<&'cache &'data str>,
}

fn main() {
    let long_lived   = String::from("I have a long life");
    let short_lived  = String::from("brief");

    // Both strings are alive here, so 'long (long_lived) outlives 'short (short_lived).
    let result = shorter_of(&long_lived, &short_lived);
    println!("shorter: {result}"); // brief
}

The 'static Lifetime

The 'static lifetime is the longest possible lifetime — it means the referenced data is valid for the entire duration of the program. There are two main ways to produce a 'static reference: string literals (&'static str literals are baked into the binary and live forever) and heap-allocated data whose ownership has been deliberately leaked with Box::leak. Additionally, types that contain no references (or only 'static references) automatically satisfy T: 'static, which is what std::thread::spawn requires for closures.

The T: 'static bound on a type does not mean the value lives forever — it means the type is free of short-lived borrows and could, in principle, be kept indefinitely. A String satisfies 'static even though it will be dropped at some point.

String literals vs heap allocations vs thread-bound values, showing what satisfies 'static and why.

use std::thread;

// Returns a &'static str: the literal is embedded in the binary.
fn static_greeting() -> &'static str {
    "Hello, static world!"
}

// Box::leak intentionally leaks a heap allocation, producing a &'static reference.
// Use sparingly -- the memory is never freed.
fn make_static_string(s: String) -> &'static str {
    Box::leak(s.into_boxed_str())
}

// A function that accepts only 'static types (common in thread/async contexts)
fn print_static<T: std::fmt::Debug + 'static>(val: T) {
    thread::spawn(move || println!("{val:?}")).join().unwrap();
}

fn main() {
    let s: &'static str = static_greeting();
    println!("{s}");

    let dynamic = String::from("I was dynamic");
    let leaked: &'static str = make_static_string(dynamic);
    println!("{leaked}");

    // String satisfies T: 'static because it owns its data (no borrows).
    print_static(String::from("owned, so 'static"));
    print_static(42_i32); // primitives are always 'static
    print_static(vec![1, 2, 3]);

    // This would NOT compile -- &str with a non-static lifetime is not 'static:
    // let local = String::from("local");
    // print_static(local.as_str()); // ERROR: `local` does not live long enough
}

Higher-Rank Trait Bounds (HRTB)

A Higher-Rank Trait Bound (HRTB) expresses a constraint that must hold for all possible lifetimes, written with the for<'a> syntax. This is necessary when a closure or function pointer must be callable with references of any lifetime, not just a specific one named in the outer scope.

The most common scenario is a function that takes a callback which operates on a reference: the callback must work regardless of the lifetime of the reference passed to it. Without HRTB, you would have to name a specific lifetime for the callback's argument, which would artificially restrict when the function can be called. With for<'a>, the bound says "for any lifetime 'a the caller might choose."

In practice, HRTB are written implicitly most of the time: when you write F: Fn(&str) -> &str, the compiler desugars this to F: for<'a> Fn(&'a str) -> &'a str automatically. You only need to write for<'a> explicitly in the rare cases where the compiler cannot infer it.

A function that maps over a slice with a closure that borrows each element — requires HRTB because the closure must accept references of any lifetime, not a fixed one.

// Without HRTB (explicit form for clarity):
// F: for<'a> Fn(&'a str) -> &'a str
// This says: F must work for ANY lifetime 'a, not just some specific 'a.
fn map_strings<F>(strings: &[String], f: F) -> Vec<String>
where
    F: for<'a> Fn(&'a str) -> &'a str,
{
    strings.iter().map(|s| f(s.as_str()).to_string()).collect()
}

// A struct holding a callback that must work with any lifetime.
struct Transformer {
    func: Box<dyn for<'a> Fn(&'a str) -> &'a str>,
}

impl Transformer {
    fn new(f: impl for<'a> Fn(&'a str) -> &'a str + 'static) -> Self {
        Transformer { func: Box::new(f) }
    }

    fn apply(&self, s: &str) -> String {
        (self.func)(s).to_string()
    }
}

fn first_word(s: &str) -> &str {
    s.split_whitespace().next().unwrap_or(s)
}

fn main() {
    let strings = vec![
        String::from("hello world"),
        String::from("rust programming"),
        String::from("higher rank"),
    ];

    // The closure |s| s must work for any lifetime -- HRTB inferred automatically here.
    let firsts = map_strings(&strings, first_word);
    println!("{firsts:?}"); // ["hello", "rust", "higher"]

    let upper = map_strings(&strings, |s: &str| -> &str {
        // Note: we can't return a reference to a local here, only to the input.
        // This is the lifetime constraint at work.
        if s.starts_with('h') { "starts_with_h" } else { s }
    });
    println!("{upper:?}");

    let t = Transformer::new(first_word);
    println!("{}", t.apply("higher rank trait bounds")); // higher
}

Variance

Variance describes how subtyping relationships between types propagate through generic containers. In Rust, this is most visible with lifetimes: if 'long is a subtype of 'short (i.e., 'long outlives 'short), does Container<'long> become a subtype of Container<'short>?

  • Covariant (e.g., &'a T, Vec<T>): a longer lifetime can be used where a shorter one is expected. Safe for read-only types.
  • Contravariant (e.g., function arguments): a shorter lifetime can be used where a longer one is expected.
  • Invariant (e.g., &'a mut T): the lifetime must match exactly. Mutable references are invariant to prevent subtle aliasing bugs.

In practice, the compiler infers variance correctly for all standard types. You only need to think about variance explicitly when designing types with raw pointers or unsafe code, or when diagnosing surprising lifetime errors involving mutable references.

A demonstration of invariance in mutable references: why you cannot substitute a longer-lived mutable reference where a shorter-lived one is expected, even though it seems safe.

// Variance in action: &mut T is INVARIANT in T's lifetime.
// This prevents a class of subtle bugs.

fn replace_with_static(r: &mut &'static str) {
    *r = "static string from binary";
}

fn main() {
    // This would be unsound if &mut &T were covariant:
    //
    // let mut local = String::from("local string");
    // let mut r: &str = &local;
    // replace_with_static(&mut r);
    // drop(local);
    // println!("{r}"); // r now dangles!
    //
    // The compiler rejects this because &mut &'a str is invariant in 'a:
    // 'a must be exactly 'static, not "at least 'static".

    // What IS valid: the mutable reference target already has 'static lifetime.
    let mut s: &'static str = "original";
    replace_with_static(&mut s);
    println!("{s}"); // static string from binary

    // Covariance in shared references: this DOES work.
    // &'long T is a subtype of &'short T (can use longer lifetime where shorter is needed).
    let long_lived = String::from("long-lived");
    let r: &str = &long_lived; // covariance: &String coerces to &str freely
    println!("{r}");
}

Practical Patterns with Advanced Lifetimes

Self-Referential Structs

Rust makes self-referential structs (a struct that contains both data and a reference to that same data) very difficult to express safely with standard lifetimes. The fundamental issue is that if the struct moves, the reference would dangle. The idiomatic solutions are: store indices instead of references, use Rc/Arc to share ownership, or for truly pinned self-referential types, use the Pin API (which appears in the async context). Understanding why they are hard clarifies a lot about what lifetimes actually guarantee.

Lifetime Bounds on Trait Objects

When storing a trait object in a struct or returning it from a function, you often need to specify a lifetime bound on the trait object itself. Box<dyn Trait + 'static> means the concrete type satisfies 'static (it owns all its data). Box<dyn Trait + 'a> allows the concrete type to hold references with lifetime 'a. The default for boxed trait objects is 'static; the default for &dyn Trait is the lifetime of the reference.

A processor that holds trait objects with explicit lifetime bounds, allowing borrowed data in implementations.

trait Processor {
    fn process(&self, input: &str) -> String;
}

// This implementation borrows data from outside -- not 'static.
struct PrefixProcessor<'a> {
    prefix: &'a str,
}

impl<'a> Processor for PrefixProcessor<'a> {
    fn process(&self, input: &str) -> String {
        format!("{}: {}", self.prefix, input)
    }
}

// An owned implementation -- satisfies 'static.
struct UpperProcessor;
impl Processor for UpperProcessor {
    fn process(&self, input: &str) -> String { input.to_uppercase() }
}

// Pipeline holds processors that may borrow data with lifetime 'a.
struct Pipeline<'a> {
    steps: Vec<Box<dyn Processor + 'a>>,
}

impl<'a> Pipeline<'a> {
    fn new() -> Self { Pipeline { steps: Vec::new() } }

    fn add_step(&mut self, step: impl Processor + 'a) {
        self.steps.push(Box::new(step));
    }

    fn run(&self, input: &str) -> Vec<String> {
        self.steps.iter().map(|s| s.process(input)).collect()
    }
}

fn main() {
    let tag = String::from("INFO");
    let mut pipeline: Pipeline<'_> = Pipeline::new();

    // PrefixProcessor borrows `tag` -- lifetime tied to `tag`'s scope.
    pipeline.add_step(PrefixProcessor { prefix: &tag });
    // UpperProcessor owns nothing -- satisfies any lifetime.
    pipeline.add_step(UpperProcessor);

    let results = pipeline.run("system started");
    for r in results {
        println!("{r}");
    }
    // INFO: system started
    // SYSTEM STARTED
}
