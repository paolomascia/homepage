Traits: Definition, Implementation & Default Methods

Rust Traits Generics

Traits are Rust's answer to the question: how do you share behaviour across unrelated types without inheritance? A trait defines a contract — a set of methods that any type implementing the trait must provide. This single mechanism covers what other languages spread across interfaces, abstract classes, type classes, and protocols. Understanding traits deeply is the key to writing idiomatic, composable, zero-cost Rust.

Defining and Implementing Traits

A trait definition lists method signatures. It says: "any type that wants to be called X must be able to do these things." The impl TraitName for TypeName block is the explicit declaration that a type fulfils that contract. Rust never applies trait implementations implicitly — every implementation is visible in the source, making it straightforward to search for all types that implement a given trait.

The most important property of trait implementations in Rust is the orphan rule: you can only implement a trait for a type if either the trait or the type is defined in your crate. You cannot implement Display for Vec<T> in your own crate, because both are from the standard library. This rule prevents two libraries from providing conflicting implementations and makes the system globally coherent.

A Basic Trait

A Summary trait with a required method, implemented for two unrelated types, then called through a shared function that accepts any type implementing the trait.

trait Summary {
    fn summarize(&self) -> String;
}

struct Article {
    title:  String,
    author: String,
    body:   String,
}

struct Tweet {
    username: String,
    content:  String,
}

impl Summary for Article {
    fn summarize(&self) -> String {
        format!("{}, by {} — {}...", self.title, self.author, &self.body[..50.min(self.body.len())])
    }
}

impl Summary for Tweet {
    fn summarize(&self) -> String {
        format!("{}: {}", self.username, self.content)
    }
}

// `impl Trait` syntax: accepts any type that implements Summary
fn notify(item: &impl Summary) {
    println!("Breaking news! {}", item.summarize());
}

fn main() {
    let article = Article {
        title:  "Rust 2026 Edition".to_string(),
        author: "Paolo Mascia".to_string(),
        body:   "The new edition brings many ergonomic improvements...".to_string(),
    };
    let tweet = Tweet {
        username: "rustlang".to_string(),
        content:  "Exciting things coming in the next release!".to_string(),
    };

    notify(&article);
    notify(&tweet);
}

Default Method Implementations

A trait can provide a default implementation for any of its methods. Types implementing the trait can use the default as-is, or override it with their own version. This is the primary mechanism for sharing common behaviour across multiple types without code duplication, and without the complexity of inheritance. Default methods can call other methods in the same trait, even required ones — this is how the standard library's Iterator trait provides dozens of adapters built entirely on top of the single required next() method.

A Printable trait with a required format_output method and a default print method that calls it, showing how some types use the default and others override it.

trait Printable {
    // Required: implementors must provide this.
    fn format_output(&self) -> String;

    // Default: calls the required method; implementors can override.
    fn print(&self) {
        println!("{}", self.format_output());
    }

    fn print_with_border(&self) {
        let s = self.format_output();
        let border = "-".repeat(s.len() + 4);
        println!("{border}");
        println!("| {s} |");
        println!("{border}");
    }
}

struct Temperature(f64);
struct Humidity(f64);
struct Pressure(f64);

impl Printable for Temperature {
    fn format_output(&self) -> String { format!("{:.1}°C", self.0) }
    // Uses default `print` and `print_with_border`
}

impl Printable for Humidity {
    fn format_output(&self) -> String { format!("{:.0}% RH", self.0) }

    // Override the default print behaviour
    fn print(&self) {
        println!("Humidity reading: {}", self.format_output());
    }
}

impl Printable for Pressure {
    fn format_output(&self) -> String { format!("{:.1} hPa", self.0) }
}

fn main() {
    let t = Temperature(23.5);
    let h = Humidity(65.0);
    let p = Pressure(1013.25);

    t.print();             // 23.5°C
    h.print();             // Humidity reading: 65% RH  (overridden)
    p.print_with_border(); // bordered 1013.2 hPa
}

Trait Bounds

A trait bound is a constraint on a generic type parameter, expressing "this parameter must implement the given trait." Trait bounds are the bridge between generics and traits: they let you write a single generic function that works with many types while still calling trait methods on them. Without bounds, the compiler refuses to call any methods on a generic type, because it has no guarantee those methods exist.

There are three equivalent syntaxes for trait bounds, each useful in different contexts:

  • impl Trait in argument position: concise, good for simple cases
  • T: Trait in angle brackets: explicit, good for multiple parameters
  • where T: Trait clause: cleanest for complex bounds with many parameters

Three equivalent ways to write the same bounded generic function, plus multiple bounds using + syntax.

use std::fmt;

// Syntax 1: `impl Trait` -- simplest
fn print_item(item: &impl fmt::Display) {
    println!("{item}");
}

// Syntax 2: explicit type parameter bound
fn print_item2<T: fmt::Display>(item: &T) {
    println!("{item}");
}

// Syntax 3: where clause -- best for multiple or complex bounds
fn print_item3<T>(item: &T) where T: fmt::Display {
    println!("{item}");
}

// Multiple bounds with `+`
fn print_and_debug<T: fmt::Display + fmt::Debug>(item: &T) {
    println!("Display: {item}");
    println!("Debug:   {item:?}");
}

// Multiple type parameters with different bounds
fn compare_and_display<T, U>(t: &T, u: &U)
where
    T: fmt::Display + PartialOrd,
    U: fmt::Display,
{
    println!("t={t}, u={u}");
}

fn main() {
    print_item(&42);
    print_item(&"hello");
    print_and_debug(&vec![1, 2, 3]);
    compare_and_display(&10, &"world");
}

Associated Types

An associated type is a type placeholder defined inside a trait and filled in by each implementation. It is similar to a generic type parameter on the trait, but with one key difference: a type can implement a trait with generics multiple times (once per type argument), but can only implement a trait with an associated type once. This makes associated types the right tool when there is a single canonical output type for a given implementor.

The Iterator trait's Item is the canonical example: a Vec<i32> iterator always yields i32; there is no reason to allow it to yield different types depending on context. An associated type expresses this "one type per implementation" constraint precisely and makes call sites cleaner — you do not have to specify the type argument everywhere the trait is used.

A Converter trait with an associated output type, implemented for two types that produce different outputs.

trait Converter {
    type Output;
    fn convert(&self) -> Self::Output;
}

struct Celsius(f64);
struct Fahrenheit(f64);

impl Converter for Celsius {
    type Output = Fahrenheit;
    fn convert(&self) -> Fahrenheit {
        Fahrenheit(self.0 * 9.0 / 5.0 + 32.0)
    }
}

impl Converter for Fahrenheit {
    type Output = Celsius;
    fn convert(&self) -> Celsius {
        Celsius((self.0 - 32.0) * 5.0 / 9.0)
    }
}

// Using the associated type in a bound
fn convert_and_print<T>(val: &T)
where
    T: Converter,
    T::Output: std::fmt::Debug,
{
    println!("{:?}", val.convert());
}

fn main() {
    let boiling = Celsius(100.0);
    let f = boiling.convert();
    println!("{:.1}°F", f.0); // 212.0°F

    let body_temp = Fahrenheit(98.6);
    let c = body_temp.convert();
    println!("{:.1}°C", c.0); // 37.0°C
}

Trait Inheritance and Supertraits

A trait can require that implementors also implement one or more other traits. This is called a supertrait relationship and is expressed with trait Child: Parent. It means: to implement Child, a type must also implement Parent. This is not inheritance in the object-oriented sense — there is no method resolution order or field sharing. It is purely a constraint: implementing Child implies implementing Parent, and methods of Parent are available in the body of Child's default methods.

A PrettyPrint supertrait that requires fmt::Display, and a blanket implementation for anything that implements PrettyPrint.

use std::fmt;

// Supertrait: implementing Animal requires implementing fmt::Display
trait Animal: fmt::Display {
    fn name(&self) -> &str;
    fn sound(&self) -> &str;

    // Default method can call supertrait methods (Display here)
    fn describe(&self) {
        println!("{} says '{}'", self, self.sound());
    }
}

struct Dog { name: String }
struct Cat { name: String }

impl fmt::Display for Dog {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "Dog({})", self.name)
    }
}
impl Animal for Dog {
    fn name(&self) -> &str { &self.name }
    fn sound(&self) -> &str { "woof" }
}

impl fmt::Display for Cat {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "Cat({})", self.name)
    }
}
impl Animal for Cat {
    fn name(&self) -> &str { &self.name }
    fn sound(&self) -> &str { "meow" }
}

fn main() {
    let dog = Dog { name: "Rex".to_string() };
    let cat = Cat { name: "Whiskers".to_string() };
    dog.describe(); // Dog(Rex) says 'woof'
    cat.describe(); // Cat(Whiskers) says 'meow'
    println!("{dog}"); // Dog(Rex)  -- Display from supertrait
}

Blanket Implementations

A blanket implementation implements a trait for every type that satisfies some bound, rather than for a specific named type. The standard library uses this extensively: for example, impl<T: Display> ToString for T gives every type that implements Display a free to_string() method. Blanket implementations are written with a generic parameter in the impl block instead of a concrete type name.

This is a powerful technique for providing convenience methods to an entire family of types at once. The key constraint is the orphan rule: a blanket implementation of a foreign trait for a foreign bound is not allowed.

A custom Describable trait with a blanket implementation for any type that implements both Display and Debug.

use std::fmt;

trait Describable {
    fn describe(&self) -> String;
}

// Blanket impl: every type that is both Display and Debug gets Describable for free.
impl<T: fmt::Display + fmt::Debug> Describable for T {
    fn describe(&self) -> String {
        format!("display='{}', debug='{:?}'", self, self)
    }
}

fn print_description(item: &impl Describable) {
    println!("{}", item.describe());
}

fn main() {
    print_description(&42_i32);
    print_description(&"hello");
    print_description(&vec![1, 2, 3]);
    print_description(&3.14_f64);
}

The Standard Library's Core Traits

The standard library defines a small set of foundational traits that the language itself is built around. Understanding these is essential for writing idiomatic Rust:

Iterator

The Iterator trait is perhaps the most important trait in the standard library. Requiring only next(&mut self) -> Option<Self::Item>, it grants access to all iterator adapter methods. Any type can become an iterator by implementing this one method.

From and Into

From<T> and Into<T> are reciprocal conversion traits. Implementing From<A> for B automatically provides Into<B> for A. They enable the idiomatic .into() call and are used heavily with error handling (? uses From for error conversion).

Clone and Copy

Clone enables explicit deep duplication with .clone(). Copy is a marker trait (no methods) that tells the compiler values can be duplicated with a bitwise copy, making them behave like primitives. Copy requires Clone and can only be applied to types whose all fields are also Copy.

PartialEq, Eq, PartialOrd, Ord

These four traits govern comparison operators. PartialEq allows == and !=; Eq additionally asserts that equality is total (reflexive, symmetric, transitive) — required for use as a HashMap key. PartialOrd allows <, > etc.; Ord requires a total ordering and is required for sort().

Display and Debug

Display (for {}) is for human-readable output. Debug (for {:?}) is for developer-readable output and must be implemented on all error types. Almost always derived with #[derive(Debug)].

A comprehensive example implementing several standard traits for a custom Matrix2x2 type, demonstrating how each trait integrates with the language.

use std::fmt;
use std::ops::{Add, Mul, Neg};

#[derive(Debug, Clone, Copy, PartialEq)]
struct Matrix2x2 {
    a: f64, b: f64,
    c: f64, d: f64,
}

impl Matrix2x2 {
    fn new(a: f64, b: f64, c: f64, d: f64) -> Self { Matrix2x2 { a, b, c, d } }
    fn identity() -> Self { Matrix2x2::new(1.0, 0.0, 0.0, 1.0) }
    fn det(&self) -> f64 { self.a * self.d - self.b * self.c }
    fn transpose(&self) -> Self { Matrix2x2::new(self.a, self.c, self.b, self.d) }
}

impl fmt::Display for Matrix2x2 {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "[[{}, {}], [{}, {}]]", self.a, self.b, self.c, self.d)
    }
}

impl Add for Matrix2x2 {
    type Output = Matrix2x2;
    fn add(self, rhs: Matrix2x2) -> Matrix2x2 {
        Matrix2x2::new(self.a + rhs.a, self.b + rhs.b, self.c + rhs.c, self.d + rhs.d)
    }
}

impl Mul for Matrix2x2 {
    type Output = Matrix2x2;
    fn mul(self, rhs: Matrix2x2) -> Matrix2x2 {
        Matrix2x2::new(
            self.a * rhs.a + self.b * rhs.c,
            self.a * rhs.b + self.b * rhs.d,
            self.c * rhs.a + self.d * rhs.c,
            self.c * rhs.b + self.d * rhs.d,
        )
    }
}

impl Default for Matrix2x2 {
    fn default() -> Self { Matrix2x2::identity() }
}

fn main() {
    let m = Matrix2x2::new(1.0, 2.0, 3.0, 4.0);
    let n = Matrix2x2::new(5.0, 6.0, 7.0, 8.0);

    println!("m = {m}");
    println!("n = {n}");
    println!("m + n = {}", m + n);
    println!("m * n = {}", m * n);
    println!("det(m) = {}", m.det()); // -2
    println!("m^T = {}", m.transpose());
    println!("identity = {}", Matrix2x2::default());
    println!("m == m.clone(): {}", m == m.clone()); // true (Copy makes this a copy)
}

Newtype Pattern and Trait Coherence

The orphan rule sometimes prevents you from implementing the trait you want for a type you do not own. The standard workaround is the newtype pattern: wrap the foreign type in a local single-field tuple struct. The wrapper is defined in your crate, so you can implement any trait for it. The overhead is zero: the wrapper is a transparent compile-time construct with no runtime cost.

A Wrapper(Vec<String>) newtype that circumvents the orphan rule to implement Display for a foreign type.

use std::fmt;

// We want to implement Display for Vec<String>, but both Vec and Display
// are from the standard library, so the orphan rule forbids it.
// Solution: wrap Vec<String> in a local newtype.
struct Wrapper(Vec<String>);

impl fmt::Display for Wrapper {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "[{}]", self.0.join(", "))
    }
}

// Implement Deref so Wrapper behaves like Vec<String> for most operations
impl std::ops::Deref for Wrapper {
    type Target = Vec<String>;
    fn deref(&self) -> &Vec<String> { &self.0 }
}

fn main() {
    let w = Wrapper(vec!["one".to_string(), "two".to_string(), "three".to_string()]);
    println!("{w}");        // [one, two, three]
    println!("{}", w.len()); // 3  -- via Deref
    println!("{}", w.join(" | ")); // one | two | three  -- via Deref
}
