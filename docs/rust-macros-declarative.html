Declarative Macros: macro_rules!

Rust Macros Metaprogramming

1. What are Macros and Why Use Them?

Rust macros operate at the syntax level — they transform token trees before the compiler type-checks anything. This gives them powers that functions cannot have:

  • Variadic arguments (println!("{}", a, b, c))
  • Generating repetitive boilerplate (vec![1, 2, 3])
  • Domain-specific mini-languages (SQL, HTML templates)
  • Implementing traits for many types at once

Rust has two kinds of macros: declarative (macro_rules!) and procedural (proc macros). This article covers declarative macros.

2. Basic macro_rules!

A macro_rules! macro is a pattern-matching rule set. Each arm matches a token pattern and expands to code.

// Simplest possible macro
macro_rules! say_hello {
    () => {
        println!("Hello, world!");
    };
}

fn main() {
    say_hello!(); // expands to: println!("Hello, world!");
}

2.1 Matching Literal Tokens

macro_rules! greet {
    (english) => { println!("Hello!"); };
    (italian) => { println!("Ciao!"); };
    (spanish) => { println!("¡Hola!"); };
}

fn main() {
    greet!(english); // Hello!
    greet!(italian); // Ciao!
    greet!(spanish); // ¡Hola!
}

2.2 Multiple Patterns per Macro

macro_rules! assert_eq_approx {
    ($left:expr, $right:expr) => {
        assert_eq_approx!($left, $right, 1e-6)
    };
    ($left:expr, $right:expr, $eps:expr) => {
        let (l, r) = ($left, $right);
        assert!(
            (l - r).abs() < $eps,
            "assertion failed: |{} - {}| = {} >= {}",
            l, r, (l - r).abs(), $eps
        );
    };
}

fn main() {
    assert_eq_approx!(1.0_f64, 1.000_000_001); // uses default eps
    assert_eq_approx!(3.14, 3.15, 0.02);        // custom eps
    // assert_eq_approx!(1.0, 2.0); // would panic
}

3. Fragment Specifiers

Each macro variable $name:kind captures a specific syntactic category. The most important ones:

Specifier Matches Example
exprany expression1 + 2, vec.len()
identidentifier or keywordmy_var, HashMap
tytypeVec<u32>, &str
patpatternSome(x), (a, b)
stmtstatementlet x = 1;
blockblock expression{ ... }
pathpathstd::io::Error
literalliteral value42, "hello"
ttany single token treecatch-all
metaattribute contentderive(Debug)
 
// Using different fragment specifiers
macro_rules! describe {
    // expr: capture any expression
    (expr: $e:expr) => {
        println!("expression evaluates to: {}", $e);
    };
    // ident: capture an identifier
    (var: $name:ident = $val:expr) => {
        let $name = $val;
        println!("created variable '{}' = {}", stringify!($name), $name);
    };
    // ty: capture a type
    (type: $t:ty) => {
        println!("type size: {} bytes", std::mem::size_of::<$t>());
    };
}

fn main() {
    describe!(expr: 2 + 2 * 3);    // expression evaluates to: 8
    describe!(var: answer = 42);    // created variable 'answer' = 42
    describe!(type: (u64, f32));    // type size: 12 bytes
}

4. Repetition — $(…)* and $(…)+

The most powerful feature of macro_rules!: match zero or more (*), one or more (+), or zero or one (?) repetitions.

// Reimplementing vec! from scratch
macro_rules! my_vec {
    // Empty case
    () => { Vec::new() };
    // One or more elements separated by commas
    // $( ... ),+ means: one or more repetitions, separated by commas
    ( $( $elem:expr ),+ $(,)? ) => {{
        // The outer block is needed to allow multiple statements
        let mut v = Vec::new();
        $( v.push($elem); )+ // expand one push per element
        v
    }};
}

fn main() {
    let a: Vec<i32> = my_vec![];
    let b = my_vec![1, 2, 3];
    let c = my_vec![10, 20, 30,]; // trailing comma: $(,)? handles it
    println!("{a:?}"); // []
    println!("{b:?}"); // [1, 2, 3]
    println!("{c:?}"); // [10, 20, 30]
}

4.1 HashMap Literal

use std::collections::HashMap;

macro_rules! hashmap {
    // key => value pairs separated by commas
    ( $( $key:expr => $val:expr ),* $(,)? ) => {{
        let mut m = HashMap::new();
        $( m.insert($key, $val); )*
        m
    }};
}

fn main() {
    let scores: HashMap<&str, u32> = hashmap! {
        "Alice" => 95,
        "Bob"   => 87,
        "Carol" => 92,
    };
    println!("{scores:?}");
}

4.2 Counting with Repetition

// Count the number of arguments at compile time
macro_rules! count {
    () => { 0usize };
    ($head:tt $($tail:tt)*) => { 1usize + count!($($tail)*) };
}

// Fixed-size array initializer
macro_rules! array {
    ( $( $elem:expr ),* $(,)? ) => {
        {
            const N: usize = count!($($elem)*);
            let arr: [_; N] = [$($elem),*];
            arr
        }
    };
}

fn main() {
    println!("count = {}", count!(a b c d)); // 4

    let arr = array![10, 20, 30, 40];
    println!("{arr:?}"); // [10, 20, 30, 40]
    // type is [i32; 4] — known at compile time
}

5. Recursive Macros

Macros can call themselves, enabling recursive patterns like building expression trees or nested data structures.

// Build nested Option types: some!(some!(none!())) = Some(Some(None))
macro_rules! none {
    () => { None };
}
macro_rules! some {
    ($e:expr) => { Some($e) };
}

// Recursive JSON-like value builder
macro_rules! json {
    // null
    (null) => { serde_like::Value::Null };
    // boolean
    (true)  => { serde_like::Value::Bool(true) };
    (false) => { serde_like::Value::Bool(false) };
    // number
    ($n:literal) => { serde_like::Value::Number($n as f64) };
    // string
    ($s:expr) => { serde_like::Value::String($s.to_string()) };
}

// Simpler example: a recursive sum macro
macro_rules! sum {
    ($x:expr) => { $x };
    ($x:expr, $($rest:expr),+) => { $x + sum!($($rest),+) };
}

fn main() {
    println!("{}", sum!(1));           // 1
    println!("{}", sum!(1, 2));        // 3
    println!("{}", sum!(1, 2, 3, 4)); // 10
}

5.1 TT Muncher Pattern

A TT muncher processes a token sequence one token at a time using recursion — the standard technique for arbitrary token parsing.

// Print each argument on its own line with its index
macro_rules! indexed_print {
    // Base case: nothing left
    (@inner $idx:expr ;) => {};
    // Recursive case: process one item then recurse
    (@inner $idx:expr ; $head:expr $(, $tail:expr)*) => {
        println!("[{}] = {}", $idx, $head);
        indexed_print!(@inner $idx + 1 ; $($tail),*);
    };
    // Entry point
    ($($items:expr),+ $(,)?) => {
        indexed_print!(@inner 0usize ; $($items),+);
    };
}

fn main() {
    indexed_print!("alpha", "beta", "gamma");
    // [0] = alpha
    // [1] = beta
    // [2] = gamma
}

The @inner prefix is a convention to mark internal "helper arms" that callers should not use directly.

6. Macro Hygiene

Rust macros are hygienic: identifiers introduced inside a macro expansion do not accidentally clash with identifiers in the caller's scope. This prevents subtle bugs common in C macros.

macro_rules! swap {
    ($a:expr, $b:expr) => {{
        // `tmp` is hygienically scoped — it cannot conflict
        // with any `tmp` variable the caller might have
        let tmp = $a;
        $a = $b;
        $b = tmp;
    }};
}

fn main() {
    let tmp = "I am the caller's tmp"; // same name, different variable
    let mut x = 1;
    let mut y = 2;
    swap!(x, y);
    println!("x={x}, y={y}"); // x=2, y=1
    println!("{tmp}"); // still "I am the caller's tmp" — no clash!
}

6.1 Breaking Hygiene on Purpose

Sometimes you want a macro to introduce an identifier into the caller's scope (e.g., a macro that declares a variable the caller can use). Pass the identifier as an argument:

macro_rules! let_timer {
    ($name:ident) => {
        // The ident comes FROM the caller, so it lives in caller's scope
        let $name = std::time::Instant::now();
    };
}

macro_rules! elapsed {
    ($name:ident) => {
        println!("{} elapsed: {:?}", stringify!($name), $name.elapsed());
    };
}

fn main() {
    let_timer!(t);
    // ... do work ...
    std::thread::sleep(std::time::Duration::from_millis(10));
    elapsed!(t); // t is visible here because it was introduced with the caller's ident
}

7. Exporting Macros

7.1 #[macro_export]

To make a macro available to users of your crate, annotate it with #[macro_export]. This places it at the crate root regardless of where it is defined.

// In src/lib.rs of crate `mylib`

#[macro_export]
macro_rules! my_assert {
    ($cond:expr) => {
        if !$cond {
            panic!("assertion failed: {}", stringify!($cond));
        }
    };
    ($cond:expr, $msg:literal) => {
        if !$cond {
            panic!("{}", $msg);
        }
    };
}

// Users can import it with:
// use mylib::my_assert;
// OR with the 2018 macro import: #[macro_use] extern crate mylib;

7.2 Macros in Modules

// Macros are not automatically in scope from sub-modules.
// Use #[macro_use] on the module to hoist them:
#[macro_use]
mod macros {
    macro_rules! log_debug {
        ($($arg:tt)*) => {
            #[cfg(debug_assertions)]
            println!("[DEBUG] {}", format!($($arg)*));
        };
    }
}

fn main() {
    log_debug!("x = {}, y = {}", 1, 2); // only in debug builds
}

8. Practical Patterns

8.1 Implementing a Trait for Multiple Types

trait Describable {
    fn describe(&self) -> String;
}

// Without macro: must write impl for each type separately
// With macro: one call covers many types
macro_rules! impl_describable {
    ($($t:ty),+) => {
        $(
            impl Describable for $t {
                fn describe(&self) -> String {
                    format!("{} ({})", self, std::any::type_name::<$t>())
                }
            }
        )+
    };
}

impl_describable!(u8, u16, u32, u64, i32, f64);

fn print_desc(d: &dyn Describable) { println!("{}", d.describe()); }

fn main() {
    print_desc(&42u32);  // 42 (u32)
    print_desc(&3.14f64); // 3.14 (f64)
}

8.2 Builder Pattern with Macros

macro_rules! make_builder {
    (
        struct $name:ident {
            $( $field:ident : $ty:ty ),* $(,)?
        }
    ) => {
        struct $name {
            $( $field: $ty, )*
        }

        paste::paste! {
            struct [<$name Builder>] {
                $( $field: Option<$ty>, )*
            }

            impl [<$name Builder>] {
                fn new() -> Self {
                    Self { $( $field: None, )* }
                }

                $(
                    fn $field(mut self, v: $ty) -> Self {
                        self.$field = Some(v);
                        self
                    }
                )*

                fn build(self) -> Result<$name, String> {
                    Ok($name {
                        $(
                            $field: self.$field.ok_or_else(||
                                format!("missing field: {}", stringify!($field))
                            )?,
                        )*
                    })
                }
            }
        }
    };
}

// Simpler version without paste crate:
macro_rules! field_method {
    ($field:ident : $ty:ty) => {
        pub fn $field(mut self, v: $ty) -> Self {
            self.$field = Some(v);
            self
        }
    };
}

8.3 Error Enum Generator

macro_rules! define_errors {
    ( $( $variant:ident ($msg:literal) ),* $(,)? ) => {
        #[derive(Debug)]
        enum AppError {
            $( $variant(String), )*
        }

        impl std::fmt::Display for AppError {
            fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                match self {
                    $( AppError::$variant(detail) =>
                        write!(f, concat!($msg, ": {}"), detail), )*
                }
            }
        }

        impl std::error::Error for AppError {}

        // Constructor functions
        impl AppError {
            $(
                #[allow(non_snake_case)]
                fn $variant(detail: impl Into<String>) -> Self {
                    AppError::$variant(detail.into())
                }
            )*
        }
    };
}

define_errors! {
    NotFound("resource not found"),
    Unauthorized("authentication required"),
    ParseError("failed to parse input"),
}

fn find_user(id: u32) -> Result<String, AppError> {
    if id == 0 {
        Err(AppError::NotFound(format!("user id {id}")))
    } else {
        Ok(format!("user-{id}"))
    }
}

fn main() {
    match find_user(0) {
        Ok(u)  => println!("found: {u}"),
        Err(e) => println!("error: {e}"), // error: resource not found: user id 0
    }
}

9. Debugging Macros

// stringify!: turns tokens into a string literal (useful for debugging)
macro_rules! dbg_expr {
    ($e:expr) => {{
        let val = $e;
        println!("{} = {:?}", stringify!($e), val);
        val
    }};
}

fn main() {
    let x = dbg_expr!(2 + 2 * 3); // 2 + 2 * 3 = 8
    let v = dbg_expr!(vec![1, 2, 3].iter().sum::<i32>()); // vec![...] = 6
    println!("{x}, {v}");
}

// Trace macro expansion: cargo rustc -- -Z macro-backtrace
// Or use: cargo expand (install: cargo install cargo-expand)
// cargo expand will show the fully expanded code

The cargo expand subcommand (from cargo-expand) shows the exact code a macro produces — invaluable for debugging complex macros.

10. Limitations of macro_rules!

  • No type inspection — macros work on tokens, not types. Use proc macros if you need type information.
  • Error messages can be cryptic. Prefer small, focused macros and test them independently.
  • Ordering — a macro_rules! macro must be defined before its first use in the same file (unlike functions or types).
  • No first-class values — macros cannot be passed as arguments or stored in variables.
  • Ambiguity restrictions — certain fragment specifiers impose restrictions on what can follow them to prevent ambiguity.

When macro_rules! is not expressive enough, reach for procedural macros — covered in the next article.
