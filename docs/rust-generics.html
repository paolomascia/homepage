Generics: Bounds, Where Clauses & Monomorphization

Rust Generics Monomorphization

Generics let you write code once and have it work across many types. In Rust, generics are a compile-time mechanism: the compiler substitutes each concrete type for the type parameter and generates specialized code with no runtime overhead. Trait bounds tell the compiler what operations are valid on a generic type, turning what would be a runtime error in other languages into a precise compile-time guarantee.

Generic Functions

A generic function is parameterized by one or more type variables declared in angle brackets after the function name. Without bounds, a type parameter is completely opaque — you cannot call any methods on it, pass it to functions, or compare it. Bounds are how you describe what you need the type to be able to do.

Rust performs monomorphization at compile time: for every unique set of type arguments a generic function is called with, the compiler generates a fully specialized, concrete version. A call to max::<i32>(a, b) and a call to max::<f64>(x, y) produce two separate functions in the binary, each with no more overhead than if you had written them by hand. This is what "zero-cost abstractions" means concretely.

Basic Generic Functions

Progressively constrained generic functions: from unbounded (virtually useless) to single-bound, to multi-bound, demonstrating what each level unlocks.

use std::fmt;

// Unbounded: T can be anything, but we can do nothing with it except store it.
fn first<T>(slice: &[T]) -> Option<&T> {
    slice.first()
}

// Single bound: PartialOrd allows `>` comparison.
fn larger<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }
}

// Multiple bounds: PartialOrd for comparison, Display for printing.
fn print_larger<T: PartialOrd + fmt::Display>(a: T, b: T) {
    let result = if a > b { a } else { b };
    println!("larger: {result}");
}

// Where clause: cleaner when bounds get complex.
fn describe_pair<T, U>(t: T, u: U) -> String
where
    T: fmt::Display + fmt::Debug,
    U: fmt::Display + Clone,
{
    format!("t={t} ({t:?}), u={u}")
}

fn main() {
    println!("{:?}", first(&[1, 2, 3]));     // Some(1)
    println!("{}", larger(3.14, 2.71));       // 3.14
    println!("{}", larger("apple", "banana")); // banana
    print_larger(100u32, 200u32);             // larger: 200
    println!("{}", describe_pair(42, "hello")); // t=42 (42), u=hello
}

Generic Structs and Enums

Structs and enums can also be generic. The standard library's most pervasive types — Option<T>, Result<T, E>, Vec<T>, HashMap<K, V> — are all generic. When you define a generic struct, you can have different impl blocks with different bounds, making certain methods available only when the type parameter satisfies specific constraints. This is called conditional implementation.

A generic Pair<T> struct with an unconditional impl block and a conditional one that adds methods only when T: PartialOrd + Display.

use std::fmt;

#[derive(Debug)]
struct Pair<T> {
    first:  T,
    second: T,
}

// Always available, regardless of T
impl<T> Pair<T> {
    fn new(first: T, second: T) -> Self {
        Pair { first, second }
    }

    fn first(&self)  -> &T { &self.first }
    fn second(&self) -> &T { &self.second }
}

// Only available when T: PartialOrd + Display
impl<T: PartialOrd + fmt::Display> Pair<T> {
    fn larger(&self) -> &T {
        if self.first >= self.second { &self.first } else { &self.second }
    }

    fn print_sorted(&self) {
        if self.first <= self.second {
            println!("({}, {})", self.first, self.second);
        } else {
            println!("({}, {})", self.second, self.first);
        }
    }
}

// Generic enum: like Option but with an additional "loading" state
#[derive(Debug)]
enum Loadable<T> {
    Loading,
    Loaded(T),
    Failed(String),
}

impl<T: fmt::Display> Loadable<T> {
    fn display(&self) {
        match self {
            Loadable::Loading     => println!("Loading..."),
            Loadable::Loaded(v)   => println!("Loaded: {v}"),
            Loadable::Failed(msg) => println!("Failed: {msg}"),
        }
    }
}

fn main() {
    let p = Pair::new(5, 10);
    println!("first: {}", p.first());
    println!("larger: {}", p.larger());
    p.print_sorted(); // (5, 10)

    let p2 = Pair::new("zebra", "apple");
    p2.print_sorted(); // (apple, zebra)

    let data: Loadable<u32> = Loadable::Loaded(42);
    data.display(); // Loaded: 42

    let err: Loadable<u32> = Loadable::Failed("timeout".to_string());
    err.display(); // Failed: timeout
}

Generic Implementations of Traits

You can implement a trait generically for a family of types. For example, you might implement a custom Summary trait for all types that implement Display, without naming any concrete type. This is called a blanket implementation and is one of the most powerful tools for code reuse in Rust. The standard library uses this technique extensively: impl<T: Display> ToString for T gives every displayable type a to_string() method without listing each one.

A generic Cache<K, V> with impl blocks that add methods depending on whether K is hashable, orderable, or displayable.

use std::collections::HashMap;
use std::fmt;

struct Cache<K, V> {
    store: HashMap<K, V>,
}

// Always available: K: Eq + Hash is required for HashMap.
impl<K: std::hash::Hash + Eq, V> Cache<K, V> {
    fn new() -> Self { Cache { store: HashMap::new() } }

    fn insert(&mut self, key: K, value: V) -> Option<V> {
        self.store.insert(key, value)
    }

    fn get(&self, key: &K) -> Option<&V> {
        self.store.get(key)
    }

    fn len(&self) -> usize { self.store.len() }
}

// Extra methods when V is also Display
impl<K: std::hash::Hash + Eq + fmt::Display, V: fmt::Display> Cache<K, V> {
    fn print_all(&self) {
        for (k, v) in &self.store {
            println!("  {k} => {v}");
        }
    }
}

fn main() {
    let mut cache: Cache<String, u32> = Cache::new();
    cache.insert("hits".to_string(), 42);
    cache.insert("misses".to_string(), 7);

    println!("hits: {:?}", cache.get(&"hits".to_string())); // Some(42)
    println!("size: {}", cache.len()); // 2
    cache.print_all(); // hits => 42 / misses => 7
}

The Monomorphization Model in Depth

Monomorphization means the compiler copies and specializes generic code for every concrete type it encounters during compilation. The process is similar to C++ template instantiation, but Rust performs it after type-checking, so errors are caught at the definition site ("this bound is not met") rather than at the instantiation site ("no member named X in class Y").

The key implication is code size: if you call sort::<i32>, sort::<f64>, and sort::<String>, you get three copies of the sort algorithm in the binary. For most code this is a non-issue and the performance benefit of inlining outweighs the size cost. For code where binary size matters (embedded, WebAssembly), you can manually introduce dynamic dispatch to reduce duplication.

A sorting utility showing how monomorphization produces type-specialized code, and a manual de-genericization approach to reduce binary size in size-constrained contexts.

// Generic sort wrapper -- monomorphized for each concrete type.
// The compiler generates three separate sort_slice functions.
fn sort_slice<T: Ord>(data: &mut [T]) {
    data.sort();
}

// Generic with key extraction
fn sort_by_key<T, K: Ord>(data: &mut Vec<T>, key: impl Fn(&T) -> K) {
    data.sort_by_key(key);
}

fn main() {
    let mut ints = vec![3, 1, 4, 1, 5, 9, 2, 6];
    sort_slice(&mut ints);
    println!("{ints:?}"); // [1, 1, 2, 3, 4, 5, 6, 9]

    let mut words = vec!["banana", "apple", "cherry", "date"];
    sort_slice(&mut words);
    println!("{words:?}"); // ["apple", "banana", "cherry", "date"]

    #[derive(Debug)]
    struct Person { name: String, age: u32 }
    let mut people = vec![
        Person { name: "Charlie".to_string(), age: 30 },
        Person { name: "Alice".to_string(), age: 25 },
        Person { name: "Bob".to_string(), age: 35 },
    ];
    sort_by_key(&mut people, |p| p.age);
    for p in &people { println!("{}: {}", p.name, p.age); }
    // Alice: 25 / Charlie: 30 / Bob: 35
}

const Generics

Since Rust 1.51, type parameters can be not just types but also constant values — integers, booleans, or characters. This is called const generics and enables truly zero-overhead fixed-size abstractions. The classic use case is fixed-size arrays: [T; N] is already parameterized by a const generic N. You can write generic functions over arrays of any compile-time-known size without heap allocation or copying into a Vec.

A stack-allocated SmallVec<T, N> that holds up to N elements without heap allocation, using const generics for the capacity parameter.

// A fixed-capacity buffer backed by a stack array.
// N is a const generic: the capacity is baked into the type at compile time.
#[derive(Debug)]
struct FixedBuffer<T, const N: usize> {
    data: [Option<T>; N],
    len: usize,
}

impl<T: Copy + Default, const N: usize> FixedBuffer<T, N> {
    fn new() -> Self {
        FixedBuffer { data: [None; N], len: 0 }
    }

    fn push(&mut self, val: T) -> bool {
        if self.len >= N { return false; }
        self.data[self.len] = Some(val);
        self.len += 1;
        true
    }

    fn pop(&mut self) -> Option<T> {
        if self.len == 0 { return None; }
        self.len -= 1;
        self.data[self.len].take()
    }

    fn len(&self) -> usize { self.len }
    fn capacity() -> usize { N }
    fn is_full(&self) -> bool { self.len == N }
}

// Generic function over all array sizes -- works with [T; 4], [T; 16], etc.
fn sum_array<const N: usize>(arr: &[i32; N]) -> i32 {
    arr.iter().sum()
}

fn main() {
    let mut buf: FixedBuffer<i32, 4> = FixedBuffer::new();
    println!("capacity: {}", FixedBuffer::<i32, 4>::capacity()); // 4

    for i in 0..5 {
        let ok = buf.push(i);
        println!("push {i}: {ok}"); // false on 5th push (capacity exceeded)
    }
    println!("len: {}", buf.len()); // 4

    while let Some(v) = buf.pop() {
        println!("pop: {v}");
    }

    let arr4:  [i32; 4] = [1, 2, 3, 4];
    let arr8:  [i32; 8] = [1, 2, 3, 4, 5, 6, 7, 8];
    println!("sum4: {}", sum_array(&arr4)); // 10
    println!("sum8: {}", sum_array(&arr8)); // 36
}

Phantom Types

A phantom type is a type parameter that appears only in a PhantomData<T> field and is not stored in the struct at all. It has zero runtime cost (PhantomData is a zero-sized type) but carries compile-time information. The canonical use case is encoding state or units of measure in the type system, so that operations mixing incompatible states or units are caught at compile time rather than at runtime.

A type-safe units system using phantom types to prevent mixing metres with kilometres or Celsius with Fahrenheit at compile time.

use std::marker::PhantomData;
use std::ops::Add;

// Unit markers -- zero-sized types used only as phantom parameters.
struct Metres;
struct Kilometres;
struct Celsius;
struct Fahrenheit;

// A measurement value tagged with its unit at the type level.
#[derive(Debug, Clone, Copy)]
struct Quantity<T, Unit> {
    value: T,
    _unit: PhantomData<Unit>,
}

impl<T, Unit> Quantity<T, Unit> {
    fn new(value: T) -> Self {
        Quantity { value, _unit: PhantomData }
    }
    fn value(&self) -> &T { &self.value }
}

// Addition is only defined for the SAME unit -- mixing is a compile error.
impl<T: Add<Output = T>, Unit> Add for Quantity<T, Unit> {
    type Output = Quantity<T, Unit>;
    fn add(self, rhs: Self) -> Self::Output {
        Quantity::new(self.value + rhs.value)
    }
}

// Explicit conversions between units
impl Quantity<f64, Metres> {
    fn to_kilometres(self) -> Quantity<f64, Kilometres> {
        Quantity::new(self.value / 1000.0)
    }
}

fn main() {
    let d1: Quantity<f64, Metres>     = Quantity::new(500.0);
    let d2: Quantity<f64, Metres>     = Quantity::new(300.0);
    let d3: Quantity<f64, Kilometres> = Quantity::new(1.5);

    let total = d1 + d2;
    println!("total metres: {:.1}", total.value()); // 800.0

    let km = d1.to_kilometres();
    println!("in km: {:.3}", km.value()); // 0.500

    // This would be a compile-time error:
    // let bad = d1 + d3; // type mismatch: Metres vs Kilometres

    let t1: Quantity<f64, Celsius>    = Quantity::new(20.0);
    let t2: Quantity<f64, Celsius>    = Quantity::new(5.0);
    // let bad2 = t1 + Quantity::<f64, Fahrenheit>::new(68.0); // compile error
    println!("temp sum: {:.1}°C", (t1 + t2).value()); // 25.0°C
}
