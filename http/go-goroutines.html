Goroutines

Go Concurrency Channels

Goroutines are lightweight execution units managed by the Go runtime, enabling concurrent code execution. The scheduler multiplexes many goroutines over fewer OS threads, making them memory-efficient. Communication typically occurs through channels, though shared-memory synchronization is also supported.

Creating Goroutines

Use the go keyword to spawn concurrent execution:

func sayHello() {
    fmt.Println("Hello")
}

go sayHello()

Important caveat: Programs may exit before goroutines finish. Use sync.WaitGroup for synchronization:

var wg sync.WaitGroup

wg.Add(1)
go func() {
    defer wg.Done()
    sayHello()
}()

wg.Wait()

Channels

Channels are typed conduits used to send and receive values between goroutines. Sends and receives block unless buffered. Basic example:

func sayHello(ch chan string) {
    ch <- "Hello"
}

func main() {
    ch := make(chan string)
    go sayHello(ch)
    msg := <-ch
    fmt.Println(msg)
}

Channel operations:

ch := make(chan int)
ch <- 42
x := <-ch
buf := make(chan int, 10)

Closing and Ranging Over Channels

Close channels to signal completion. Receivers detect closure via a second return value or range loops:

ch := make(chan int)

go func() {
    for i := 0; i < 3; i++ {
        ch <- i
    }
    close(ch)
}()

for v := range ch {
    fmt.Println("got:", v)
}

v, ok := <-ch

Only the sender should close a channel; never close a channel from the receiving side.

Directional Channels

Restrict channel directions in function signatures for type safety:

func producer(out chan<- int) {
    for i := 0; i < 3; i++ {
        out <- i
    }
    close(out)
}

func consumer(in <-chan int) {
    for v := range in {
        fmt.Println("consumed:", v)
    }
}

Select Statement

The select statement waits on multiple channel operations, executing whichever is ready first:

select {
case msg1 := <-ch1:
    fmt.Println("Received message 1", msg1)
case msg2 := <-ch2:
    fmt.Println("Received message 2", msg2)
default:
    fmt.Println("No message available")
}

Timeouts

select {
case v := <-ch:
    fmt.Println("value:", v)
case <-time.After(500 * time.Millisecond):
    fmt.Println("timeout")
}

Panic / Recover

panic aborts normal control flow for unrecoverable errors or programmer bugs. recover regains control in deferred functions within the same goroutine:

func main() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered from panic:", r)
        }
    }()

    fmt.Println("Start of the program")
    doSomething()
    fmt.Println("End of the program")
}

func doSomething() {
    fmt.Println("Start of doSomething function")
    panic("Panic occurred")
    fmt.Println("End of doSomething function")
}

Key points:

  • recover() only works inside a deferred function
  • A panic in one goroutine won’t be recovered by a defer in another

Common Patterns & Pitfalls

Worker Pool (Fan-Out/Fan-In)

jobs := make(chan int)
results := make(chan int)

var wg sync.WaitGroup
worker := func() {
    defer wg.Done()
    for j := range jobs {
        results <- (j * j)
    }
}

wg.Add(3)
for i := 0; i < 3; i++ {
    go worker()
}

go func() {
    for i := 1; i <= 5; i++ {
        jobs <- i
    }
    close(jobs)
}()

go func() {
    wg.Wait()
    close(results)
}()

for r := range results {
    fmt.Println("result:", r)
}

Avoiding Goroutine Leaks

Always ensure goroutines can terminate: drain or close channels, respect contexts, or provide explicit quit signals.
