Digital Certificates

Cryptography PKI X.509

Consider a scenario where Bob sends a signed message to Alice. Bob signs a hash of the message using his private key to produce a signature. Both the message and signature are transmitted to Alice, who verifies by recomputing the hash and checking the signature with Bob’s public key. Successful verification demonstrates authenticity (the signer possessed Bob’s private key) and integrity (the message was not modified).

The core challenge is ensuring Alice knows Bob’s true public key. Digital certificates solve this by binding an identity (the subject) to a public key, signed by an issuer—typically a Certification Authority (CA). Software trusts root CAs and validates certificates by building a chain of trust up to a trusted root.

Certificates on the Internet

The most common application is Transport Layer Security (TLS), where servers present certificates containing their domain name(s). Clients validate the certificate chain and check hostname matching against the Subject Alternative Name (SAN).

Certificate lifecycle management—issuance, validation, renewal, and revocation—falls under Public Key Infrastructure (PKI). Internal PKIs use self-signed roots for private use, while public web servers must chain to publicly trusted CAs for browser acceptance.

Other Common Applications

  • Secure email (S/MIME, PEM): certificates enable signing and encryption for end-to-end protection.
  • VPNs (IPsec): peers authenticate with certificates; key exchange uses IKE.
  • Payments and legacy protocols: early standards like SET protected cardholder data and authenticated parties.

X.509 Certificates (v3)

X.509 is the dominant format, containing the subject’s public key and identity attributes, signed by the issuer. Version 3 introduced extensions for constraints and intended usages.

Core Fields

  • Version and Serial Number
  • Signature Algorithm (used by issuer to sign)
  • Issuer (Distinguished Name of CA)
  • Validity (Not Before / Not After)
  • Subject (Distinguished Name of owner)
  • Subject Public Key Info (algorithm + public key)

Common v3 Extensions

  • Basic Constraints (CA=true/false, path length)
  • Key Usage (e.g., digitalSignature, keyEncipherment)
  • Extended Key Usage (e.g., serverAuth, clientAuth, emailProtection, codeSigning)
  • Subject Alternative Name (SAN) (DNS names, IPs)
  • Authority Key Identifier / Subject Key Identifier
  • CRL Distribution Points / OCSP AIA (revocation endpoints)

Path Validation

# Certificate chain validation (conceptual)
# presented_cert -> intermediate_1 -> ... -> root_ca

for each cert in chain:
  # Check signature with issuer's public key
  verify(cert.signature, issuer.public_key)

  # Check validity period
  assert cert.not_before <= now <= cert.not_after

  # Check revocation status (CRL or OCSP)
  check_revocation(cert)

  # Enforce constraints
  check_basic_constraints(cert)
  check_key_usage(cert)
  check_extended_key_usage(cert)

# Hostname check uses SAN (DNS entries)
assert hostname in cert.subject_alternative_names

Advantages

  • Privacy: enables encryption for data-in-transit (HTTPS, email).
  • Authentication: binds identities to keys, preventing impersonation when properly validated.
  • Automation & Cost: issuance and renewal can be automated (e.g., ACME); many server certificates are inexpensive or free.
  • Flexibility: usable for servers, clients, code signing, document signing, devices/IoT, and internal PKIs.

Limitations and Practical Pitfalls

  • Trust Model Complexity: security depends on CA practices and client trust stores; a compromised CA can issue fraudulent certificates.
  • Revocation Fragility: CRLs/OCSP can be slow or unavailable; soft-fail behaviors may accept revoked certificates.
  • Key Management Risks: leaked private keys (server, user, or CA) expose sessions until revocation and rollover complete.
  • Usability & Ops Overhead: provisioning, renewal, and rotation must be automated to avoid outages from expired certificates.
  • Algorithm Deprecation: older algorithms (SHA-1, RSA-1024) become unsafe; ecosystems must migrate to stronger standards.
  • Name Constraints & SAN: hostname mismatches cause failures; overly broad SANs create security gaps.
  • Phishing & DV Certs: a valid TLS padlock does not guarantee legitimacy; Domain Validation only authenticates name control.
  • Middleboxes & Inspection: enterprise TLS interception with custom roots can weaken security if mishandled.
  • Self-Signed Usage: fine for internal PKIs but untrusted on public clients; requires secure root distribution.
