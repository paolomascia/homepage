Building Domain-Specific Languages (DSLs) in Go Using Generics and Reflection

Go Generics Reflection

A Domain-Specific Language (DSL) is a small, focused language designed to solve problems in a specific domain — like configuration, validation, data pipelines, or query building. While Go is not a dynamically-typed language, its generics (since Go 1.18) and reflection capabilities make it possible to implement elegant, type-safe DSLs that feel expressive and declarative.

Why Build a DSL in Go?

Go favors explicitness, but many problems benefit from higher-level abstractions. DSLs help reduce boilerplate, enforce consistency, and let developers express intent rather than control flow.

Common use cases include:

  • Query builders (like SQL or API filters)
  • Validation frameworks
  • Declarative configuration
  • Infrastructure definitions (IaC-like systems)

Generics for Type-Safe Builders

Generics let you write reusable DSL components while preserving type safety. Here’s an example of a mini “query builder” DSL built using generics:

package main

import (
	"fmt"
	"strings"
)

type Field[T any] struct {
	Name string
}

type Condition[T any] struct {
	Field Field[T]
	Op    string
	Value T
}

func Eq[T any](f Field[T], v T) Condition[T] {
	return Condition[T]{Field: f, Op: "=", Value: v}
}

func Gt[T any](f Field[T], v T) Condition[T] {
	return Condition[T]{Field: f, Op: ">", Value: v}
}

func (c Condition[T]) String() string {
	return fmt.Sprintf("%s %s '%v'", c.Field.Name, c.Op, c.Value)
}

type Query struct {
	table      string
	conditions []string
}

func NewQuery(table string) *Query {
	return &Query{table: table}
}

func (q *Query) Where(cond fmt.Stringer) *Query {
	q.conditions = append(q.conditions, cond.String())
	return q
}

func (q *Query) Build() string {
	return fmt.Sprintf("SELECT * FROM %s WHERE %s;", q.table, strings.Join(q.conditions, " AND "))
}

func main() {
	Name := Field[string]{Name: "name"}
	Age := Field[int]{Name: "age"}

	query := NewQuery("users").
		Where(Eq(Name, "Alice")).
		Where(Gt(Age, 18)).
		Build()

	fmt.Println(query)
}

Output:

SELECT * FROM users WHERE name = 'Alice' AND age > '18';

Here, generics ensure that the types of conditions are consistent — you can’t compare a string field to an int by mistake.

Using Reflection for Dynamic DSLs

Reflection allows you to inspect and manipulate types at runtime. This can be useful for DSLs that need to serialize structs, generate configuration files, or build generic validators.

package main

import (
	"fmt"
	"reflect"
)

func PrintStruct(v any) {
	val := reflect.ValueOf(v)
	typ := reflect.TypeOf(v)

	if typ.Kind() != reflect.Struct {
		fmt.Println("Not a struct")
		return
	}

	for i := 0; i < typ.NumField(); i++ {
		field := typ.Field(i)
		value := val.Field(i)
		fmt.Printf("%s = %v\n", field.Name, value.Interface())
	}
}

type Config struct {
	Host string
	Port int
}

func main() {
	cfg := Config{Host: "localhost", Port: 8080}
	PrintStruct(cfg)
}

Output:

Host = localhost
Port = 8080

Reflection-based DSLs are powerful when you need dynamic schema inspection, data mapping, or automatic generation of configurations and documentation.

Combining Generics and Reflection

You can combine generics and reflection for even more expressive DSLs — for example, a validation engine that infers rules from struct tags:

package main

import (
	"fmt"
	"reflect"
)

type User struct {
	Name string `validate:"required"`
	Age  int    `validate:"min=18"`
}

func Validate[T any](obj T) {
	val := reflect.ValueOf(obj)
	typ := reflect.TypeOf(obj)

	for i := 0; i < typ.NumField(); i++ {
		field := typ.Field(i)
		tag := field.Tag.Get("validate")
		value := val.Field(i)

		if tag == "required" && value.IsZero() {
			fmt.Printf("Field %s is required\n", field.Name)
		}
		if tag == "min=18" && value.Int() < 18 {
			fmt.Printf("Field %s must be at least 18\n", field.Name)
		}
	}
}

func main() {
	u := User{Name: "", Age: 15}
	Validate(u)
}

Output:

Field Name is required
Field Age must be at least 18

This pattern is similar to libraries like go-playground/validator, but implemented manually to illustrate reflection-driven DSLs.

When to Use a DSL in Go

DSLs in Go make sense when:

  • The domain has repetitive, rule-based logic (validation, configuration).
  • Abstractions can improve readability without hiding complexity.
  • Strong typing helps prevent misuse or invalid state.
  • The performance cost of reflection is acceptable (for non-hot paths).

Best Practices

  • Use generics for compile-time safety and clear syntax.
  • Use reflection only for dynamic behavior where necessary.
  • Keep DSLs declarative and avoid embedding control logic.
  • Document the DSL syntax and error behavior clearly.
  • Benchmark reflection-heavy paths using go test -bench.

Example Use Cases

  • Config file parsers (JSON/YAML into structs with tags).
  • SQL or API query builders (like gorm or ent).
  • Pipeline builders (chaining transformations).
  • Code generation tools (parsing Go structs with reflection).

Conclusion

By combining Go’s modern generics with runtime reflection, you can design expressive, maintainable, and type-safe Domain-Specific Languages. While Go is intentionally not dynamic, these tools make it surprisingly capable for declarative frameworks, configuration engines, and internal DSLs that rival those of more dynamic languages.
