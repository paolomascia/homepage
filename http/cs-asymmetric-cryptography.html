Asymmetric (Public-Key) Cryptography

Cryptography Public-Key RSA

Asymmetric cryptography uses a pair of mathematically related keys: a public key, which anyone can know, and a private key, which must be kept secret. Anyone can encrypt a message to the public key; only the private key holder can decrypt. Conversely, a sender can sign with the private key so that anyone can verify authenticity with the corresponding public key. These ideas underpin TLS/HTTPS and many other protocols.

RSA

RSA (Rivest–Shamir–Adleman) is the best-known public-key scheme. In practice, RSA is used for small data (key transport, signatures), while bulk data is protected by fast symmetric ciphers (e.g., AES). RSA arithmetic lives modulo an integer n = p·q, the product of two large primes.

Core Operations

// Public key:  (n, e)
// Private key: (n, d)
// Messages are integers in {0, 1, ..., n-1} after encoding/padding.

// Encryption
y = x^e mod n

// Decryption
x = y^d mod n

Here, e is the public exponent and d the private exponent. In practice, messages are padded/encoded (e.g., OAEP for encryption; PSS for signatures) to add randomness and prevent structural attacks. Also note: a message must be an integer strictly less than n (so its bit-length is < log2 n).

Key Generation (Conceptual)

  1. Choose large random primes p and q.
  2. Compute n = p·q and Euler’s totient φ(n) = (p−1)(q−1) (or Carmichael’s λ(n)).
  3. Choose public exponent e coprime to φ(n) (common choices are small, e.g., 65537).
  4. Compute the private exponent d as the modular inverse of e modulo φ(n): d·e ≡ 1 (mod φ(n)).
# Extended Euclid finds d = e^{-1} mod phi(n)
g, x, y = egcd(e, phi_n)
assert g == 1
d = x % phi_n

Security requirements (informal): recovering d from (n, e) should be computationally infeasible; modular exponentiations xe mod n and yd mod n should be efficient; for a given n, many valid key pairs should exist.

Toy Example (Small Integers)

# Primes
p = 7
q = 11
n = p * q          # n = 77
phi_n = (p-1) * (q-1)  # phi(n) = 60

# Public exponent
e = 7               # gcd(7, 60) = 1 -> OK

# Encrypt x = 9
y = pow(9, 7, 77)   # y = 37

# Private exponent d solves: d*e = 1 (mod 60)
d = 43               # since 7*43 = 301 = 1 (mod 60)

# Decrypt
x = pow(37, 43, 77)  # x = 9

Tiny parameters are for illustration only; real RSA uses moduli of at least 2048 bits and standardized padding/encoding.

Diffie–Hellman Key Exchange (DH)

Diffie–Hellman lets two parties agree on a shared secret over a public channel. They publish common parameters: a large prime p and a generator g of a cyclic subgroup. Each side picks a secret exponent and publishes a corresponding public value; both then derive the same shared key.

# Public parameters: p (prime), g (generator)

# Alice (secret a)
A = pow(g, a, p)

# Bob (secret b)
B = pow(g, b, p)

# Shared key (both compute):
# K = B^a mod p = (g^b)^a mod p = g^(ab) mod p
# K = A^b mod p = (g^a)^b mod p = g^(ab) mod p
K_alice = pow(B, a, p)
K_bob   = pow(A, b, p)
# K_alice == K_bob

The derived K seeds a symmetric cipher (e.g., AES). Classical DH provides confidentiality of the key agreement assuming the discrete logarithm problem is hard in the chosen group. However, unauthenticated DH is vulnerable to man-in-the-middle attacks; protocols use signatures or certificates to authenticate the parties.

Toy Example (Small Numbers)

p = 29
g = 2

# Alice: a = 5
A = pow(2, 5, 29)    # A = 3

# Bob: b = 12
B = pow(2, 12, 29)   # B = 7

# Exchange A=3, B=7
K_alice = pow(B, 5, 29)   # 7^5  mod 29 = 16
K_bob   = pow(A, 12, 29)  # 3^12 mod 29 = 16

# Shared secret: 16

The Discrete Logarithm Problem (DLP)

Given a finite cyclic group (e.g., integers modulo a prime p under multiplication) and a generator g, the discrete logarithm of β is the unique exponent x such that gx ≡ β (mod p). Computing x from (g, β, p) is believed to be hard for appropriate parameters, which underlies the security of Diffie–Hellman and related schemes.
