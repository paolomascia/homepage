DoS Attacks (Part 2): SYN Flood, ICMP/UDP Floods, Fragmentation, Smurf & Amplification

Cybersecurity DoS/DDoS Protocol Attacks
Educational Disclaimer
This document is for educational and defensive cybersecurity purposes only. The techniques described should only be used in authorized penetration-testing environments. Unauthorized use may be illegal and could result in criminal charges. Always obtain written authorization before testing any systems. Use this knowledge responsibly to improve security defenses.

Slowloris and R.U.D.Y. are application-layer (OSI L7) DoS attacks. In contrast, several high-impact DoS/DDoS techniques operate at lower layers, targeting transport (L4) and network (L3) behaviors. This article explains the mechanics at a high level and focuses on defensive practices.

DoS/DDoS Examples by OSI Layer

OSI Layer Category Examples
Application (L7) App-Level Exhaustion HTTP Flood, Slowloris, R.U.D.Y.
Transport (L4) Protocol Resource Exhaustion SYN Flood, UDP Flood
Network (L3) Volume/Bandwidth Saturation ICMP Flood (Ping Flood), Fragmentation Abuse

SYN Flood (Transport Layer – L4)

How TCP Connections Start (Three-Way Handshake)

  1. Client → Server: Sends SYN with initial sequence number x.
  2. Server → Client: Replies SYN/ACK with its initial sequence number y and acknowledgment x+1.
  3. Client → Server: Sends ACK acknowledging y+1. Connection opens.

How SYN Flood Works

  1. Attacker sprays the target with a high volume of SYN packets (often with spoofed IPs).
  2. Server allocates resources and replies SYN/ACK for each half-open connection, awaiting final ACK.
  3. Final ACK never arrives; half-open slots fill up; legitimate clients are starved.

Defensive Practices

  • SYN cookies (defer allocation until handshake completes).
  • Backlog tuning and reduced timeout for half-open connections.
  • Rate limiting and per-source connection caps.
  • Stateful firewalls / ADCs / WAFs with SYN proxy/intercept features.
  • Ingress/egress filtering (BCP 38/84) to reduce spoofing on upstream networks.
  • Deploy scrubbing/mitigation services and anycast/CDN to absorb spikes.

Ping Flood (ICMP Flood – Network Layer L3)

A Ping Flood overwhelms a target with ICMP Echo Requests. Each request consumes bandwidth and CPU (and may elicit Echo Replies), saturating links and devices. To be effective, attackers often use botnets to outmatch the victim’s capacity — turning it into a DDoS.

Defensive Practices

  • Rate limit ICMP at perimeter devices; allow diagnostic ICMP selectively.
  • Prefer policing over complete blocking to retain Path MTU Discovery helpers.
  • Leverage upstream filtering/scrubbing during volumetric spikes.
  • Monitor for asymmetric ICMP surges and alert on anomalies.

UDP Flood (Transport Layer L4)

A UDP Flood targets random destination ports with UDP packets. The host must check for listeners and will often generate ICMP Destination Unreachable responses, consuming additional resources. At scale (especially via reflection/amplification), this can render services unavailable.

Defensive Practices

  • Drop or rate limit unsolicited UDP to closed ports at the edge.
  • Harden services that must remain open (strict ACLs, authentication, quotas).
  • Stateless filtering on upstream routers for obviously spoofed patterns.
  • Use DDoS scrubbing and anycast to distribute impact.

Ping of Death (Legacy Fragmentation Abuse)

The classic Ping of Death exploited ancient IP stack bugs by forcing reassembly of oversized (>65,535 bytes) packets via fragmentation, leading to crashes. Modern systems are patched; however, the principle — malformed fragmentation causing reassembly issues — still informs testing and defense.

Defensive Practices

  • Ensure systems are fully patched; enforce robust reassembly limits on firewalls.
  • Drop malformed fragments and enforce sane MTU policies.
  • Continuously test for legacy protocol behaviors in lab environments only.

Fragmented Packet Attacks (Reassembly Exhaustion)

During normal IP transmission, large datagrams are fragmented and later reassembled. Attackers can send fake/incomplete fragments that never reassemble, occupying buffers and exhausting memory/CPU on middleboxes or hosts.

Defensive Practices

  • Lower fragment reassembly timeouts and limit queues.
  • Drop overlapping or suspicious fragments; prefer asymmetric DF-bit routing with PMTUD.
  • Offload to devices that normalize traffic (fragment reassembly at the edge).

Smurf DDoS (ICMP Broadcast Amplification)

A Smurf attack sends spoofed ICMP Echo Requests to a network’s broadcast address; every host replies to the spoofed victim, amplifying traffic.

Defensive Practices

  • Disable directed broadcast on routers (prevents forwarding to broadcast addresses).
  • Configure hosts to ignore broadcast ICMP.
  • Implement source address validation (BCP 38) to reduce spoofing.
  • Use upstream scrubbing during events.

NTP Amplification (and Other UDP Amplifiers)

In NTP amplification, attackers spoof a victim’s IP and issue small queries to open NTP servers that respond with much larger payloads. The result is reflected, amplified traffic to the victim. Historically, the monlist command was a notorious vector; modern NTP hardening removes or restricts it, but similar reflection exists across many UDP services.

Known UDP Amplification Vectors (examples)

  • NTP (legacy monlist)
  • SNMPv2
  • NetBIOS
  • QOTD
  • BitTorrent / Kad
  • Quake Network Protocol
  • Steam Protocol

Defensive Practices

  • Do not expose unnecessary UDP services to the Internet.
  • Harden NTP: disable legacy monitoring features; restrict queries to trusted clients.
  • Rate limit and ACL control planes; prefer anycast/CDN fronting.
  • Adopt uRPF/BCP 38 to curb spoofed-source traffic.
  • Participate in coordinated response with ISPs and DDoS mitigation providers.

Key Takeaways

  • Lower-layer DoS attacks exploit protocol behaviors and resource limits, often at massive scale via botnets.
  • Defense is layered: protocol hardening, rate limits, anomaly detection, and upstream mitigation.
  • Prevent spoofing and unnecessary UDP exposure to shrink your attack surface.
  • Continuously test in authorized labs and keep systems patched and observed.
